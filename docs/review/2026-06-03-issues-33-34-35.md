## 🔴 BLOCKER

### B1 — #34 dead-man watchdog `os._exit(1)` ทิ้ง child claude.exe เป็น orphan
`app.py:_start_deadman_watchdog` → `os._exit(1)` ตรงๆ **ข้าม `atexit` → `_kill_all()` ไม่ทำงาน**
(`os._exit` ไม่เรียก atexit/cleanup handler ตาม spec). ตรวจแล้ว **ไม่มี Job Object** ผูกชีวิต child
(`git grep JobObject/AssignProcessToJobObject` = 0 hit) — child cleanup พึ่ง `_kill_all` ทางเดียว
(`app.py:137-150`).

ผล: บน Windows พ่อตาย → child ไม่ตายตาม (ไม่มี process-tree kill) → claude.exe + winpty-agent
5 ตัว **ค้างเป็น orphan** หลัง watchdog ฆ่า orchestrator. นี่คือ **bug class เดียวกับที่ #34 ตั้งใจแก้**
(zombie/orphan process) — แค่ย้ายจาก orchestrator ไปอยู่ที่ child แทน (ตรงกับที่ task flag ไว้เป๊ะ).

**Fix:** ก่อน `os._exit(1)` ทำ best-effort terminate children (เหมือน body ของ `_kill_all`,
ครอบ try/except). `pane.session.terminate()` เป็น OS-level (TerminateProcess) ไม่ต้องพึ่ง main thread
ที่ wedge อยู่ จึงเรียกจาก daemon thread ได้:
```python
if _watchdog_should_exit(...):
    try:
        for pane in list(window.orch.panes.values()):
            if pane.session is not None:
                try:
                    pane.mark_expected_exit(); pane.session.terminate()
                except Exception: pass
    except Exception: pass
    os._exit(1)
```

## 🟡 ควรแก้

### S1 — #34 watchdog อาจ false-kill งาน sync บน main thread ที่ใช้เวลา > 30s
30s threshold สมเหตุผลสำหรับ dialog (ดู ✅V1) แต่ **ถ้ามี subprocess.run / git clone / map-paths scan
รันแบบ synchronous บน Qt main thread** (คือ root ของ #21) ที่นานเกิน 30s → event loop หยุด →
heartbeat ค้าง → watchdog `os._exit` ฆ่ากลางคันทั้งที่งานปกติ. วิธีถูกคือย้ายงานออกจาก main thread
(ไม่ใช่ดัน threshold สูงขึ้น). **Action:** audit main-thread sync op ที่อาจ > 30s; ถ้ามี ให้ย้ายเป็น worker
ก่อนเปิด watchdog ใช้จริง (ไม่งั้น watchdog แปลง hang เป็น kill แต่ผู้ใช้เสียงานที่กำลังทำ).

#### S1 Audit Results (2026-06-03)

Scanned `app.py` + `main_window.py` for blocking calls on the Qt main thread:

| Call site | Location | Duration risk | Assessment |
|---|---|---|---|
| `try_silent_self_update()` → `subprocess.run` git pull | `app.py:173` (before watchdog starts) | up to 15 s | **Safe** — watchdog not yet running; bounded `timeout_pull=15s` |
| `subprocess.Popen` (updater spawn) | `main_window.py:2213,2220,2468` | fire-and-forget (no wait) | **Safe** — detached, no `.wait()` |
| `_RulesGeneratorThread` → subprocess.run claude headless | `main_window.py:2760` | up to ~2 min | **Safe** — runs in QThread; main thread sits in `busy.exec()` (nested event loop, heartbeat ticks) |
| `thread.wait(3000)` on cancel | `main_window.py:2774` | ≤ 3 s | **Safe** — well under 30 s |
| `thread.wait(5000)` after dialog close | `main_window.py:2785` | ≤ 5 s | **Safe** — well under 30 s |
| `p.iterdir()` in `_run_map_paths_dialog` | `main_window.py:2838` | <1 s | **Safe** — lists one directory level |

**Conclusion: No main-thread sync op > 30 s found.** Watchdog false-positive risk is negligible with current codebase. If `subprocess.run` / long I/O is ever added directly to the Qt main thread (issue #21 pattern), it must be moved to a QThread — or heartbeat must be paused — before enabling the watchdog in production.

## ✅ ตรวจแล้ว ถูกต้อง (ไม่ใช่ bug)

- **V1 — modal dialog ไม่ทำ watchdog false-positive (ประเด็นหลักที่ task กังวล):** heartbeat เป็น **QTimer**
  (`main_window.py:790`). `QDialog.exec/QMessageBox.exec` รัน **nested event loop** ซึ่ง **pump timer event ต่อ**
  → `_tick_heartbeat` ยังเดิน → `_heartbeat_ts` ขยับ → watchdog **ไม่ fire** แม้ user เปิด dialog ค้าง > 30s.
  watchdog fire เฉพาะตอน event loop **หยุดจริง** (busy-loop / subprocess.run บน main thread) ซึ่งคือเคสที่ตั้งใจจับ.
  dialog ที่ flag มา (closeEvent's QMessageBox.question, `_ask_project_description`, `_show_rules_editor_dialog`,
  `_run_map_paths_dialog`) ล้วน Qt-drawn exec → timer เดินต่อ → **ปลอดภัย**. การออกแบบนี้ถูกต้องและสง่า
  (แยก "busy แต่ event loop เดิน" ออกจาก "wedged"). *เว้นแต่* dialog ไปเรียกงาน sync บน main thread นานเกิน → ดู S1.
- **V2 — render coalesce thread-safe:** `bytesIn` emit จาก `_on_bytes` ซึ่ง **รันบน main thread** (queued จาก
  reader thread, ยืนยันด้วย comment `pty_session.py:292-297`). ⇒ `_coalesce_bytes` แตะ QTimer บน main thread =
  ถูกต้อง, buffer single-thread, ไม่ต้อง lock.
- **V3 — ordering / byte-loss:** chunk append เข้า bytearray ตามลำดับมาถึง (main thread sequential) → flush เขียนทั้งก้อนตามลำดับ ไม่สลับ. flush ก่อน disconnect ใน detach (404 ก่อน 408) → ไม่ตกหล่นตอน session-swap. cap-flush 256KB ตัดกลาง escape ได้ แต่ xterm.js buffer partial sequence ข้าม write() ได้เอง → ปลอดภัย. timer lifecycle (singleShot, start เมื่อ idle, stop ตอน flush/detach) สะอาด ไม่ leak.
- **V4 — throughput watchdog logic:** วางก่อนบรรทัด `continue` ของ content-stale (`3451`) จึงจับ pane ที่ content เปลี่ยนตลอดได้ (ช่องโหว่เดิม). Lead exempt (`3365`). first-tick guard (`tp_last_ts > 0`) กัน delta ระเบิดรอบแรก. **warn-only ไม่ auto-kill** + cooldown 5 นาที → false-positive ราคาถูกมาก (build จริงพ่น > 500KB/s ต่อเนื่อง 60s ยาก, ถึงโดนก็แค่เตือน Lead ตัดสินเอง). threshold 500KB/s · 60s สมเหตุผล.
- **V5 — #33 ไม่ token bloat / ไม่ leak secret:** inject แค่ **pointer (path) + คำสั่งให้ Read on-demand** ไม่ฝังเนื้อ memory → ไม่บวม token, ไม่มี secret เข้า prompt. path encoding (`os.sep`→`-`, `:`→`-`) ตรง convention ของ Claude Code (`C:\...` → `C--...`), ยืนยันด้วย test. *(แต่ resolve ผิด cwd — ดู B2)*

## ✅ RE-REVIEW รอบสุดท้าย (2026-06-03) — BLOCKER ปิดหมด

ตรวจ diff + test ปัจจุบัน ยืนยัน **ทั้ง 2 BLOCKER ปิดจริง ไม่มี regression**.

### B1 — ปิด ✅ (`app.py:_start_deadman_watchdog`)
- ก่อน `os._exit(1)` มี loop terminate child sessions ครบ — snapshot `list(window.orch.panes.values())` + `pane.mark_expected_exit(); pane.session.terminate()` ตรงตาม body ของ `_kill_all` (proven pattern, `app.py:156-163`).
- **กัน race จาก daemon thread:** ครอบ **2 ชั้น** — try/except ต่อ pane (inner) + try/except รอบ loop (outer). teardown บน main thread ลบ pane ระหว่าง iterate → outer catch จับ → ไม่ propagate.
- **os._exit ออกได้เสมอแม้ terminate พัง:** ยืนยันด้วย test `test_watchdog_calls_os_exit_when_heartbeat_stale` — `window` เป็น `MagicMock`, `list(window.orch.panes.values())` raise TypeError (MagicMock ไม่ iterable) → outer except กลืน → `os._exit(1)` ยังถูกเรียก (assert `exit_called == [1]` ผ่าน). นี่คือ proof ตรงๆ ว่า terminate loop ไม่มีทางบล็อก exit.
- `session.terminate()` เป็น OS-level (TerminateProcess, `pty_session.py:333-359`) เรียกจาก daemon thread ได้; thread join มี timeout 500ms → bounded ไม่ hang.

### B2 — ปิด ✅ (`orchestrator.py:1294`)
- เปลี่ยนเป็น `_resolve_project_memory(lead_cwd(project_ns) or spawn_cwd)` — resolve จาก Lead's project root ตรงกับ key ที่ auto-memory เขียน (`lead_cwd` = explicit lead key / common parent / first path, `config.py:125-161`).
- **เคส lead_cwd = None ปลอดภัย:** fallback เป็น `spawn_cwd` ซึ่ง `= cwd or lead_cwd(...) or str(REPO_ROOT)` (`orchestrator.py:1267`) → ไม่มีทางเป็น None.
- **test ครอบ multi-path จริง:** `TestMemoryResolveMultiPathProject` (3 เคส) — baseline sub-cwd → None, lead_cwd → match, + `lead_cwd or spawn_cwd` expression ตรงกับ code. ปิดทั้ง B2 และ gap ที่ S2 ชี้ (test เดิม pass cwd เดียวกันจับ bug ไม่ได้).
