## 1. Correctness

### ЁЯФ┤ MAJOR тАФ Screenshot dir mtime is project-scoped, stall detection is role-scoped

**File:** `orchestrator.py:2004-2010` (`_compute_last_progress_ts`)

```python
today = datetime.now().strftime("%Y-%m-%d")
shot_dir = RUNTIME_DIR / "exports" / today / project_ns / "screenshots"
try:
    mt = shot_dir.stat().st_mtime
    if mt > ts:
        ts = mt
except OSError:
    pass
```

`shot_dir` is `runtime/exports/<date>/<project>/screenshots/` тАФ shared across **all roles in the project**. When QA captures a screenshot, the directory mtime updates and suppresses stall detection for every other role in the same project тАФ including a genuinely stalled backend or frontend pane. A stalled `backend` in a project where `qa` is actively running screenshots will never appear as stalled.

**Suggested fix:** Either scope screenshots per-role (`тАж/<project>/<role>/screenshots/`) or remove the screenshot signal from non-QA panes (guard with `if role in ("qa", "critic", "designer")`). The latter is simpler:
```python
# Only apply screenshot signal to UI-heavy roles
if role in ("qa", "critic", "designer"):
    shot_dir = RUNTIME_DIR / "exports" / today / project_ns / "screenshots"
    try:
        mt = shot_dir.stat().st_mtime
        if mt > ts:
            ts = mt
    except OSError:
        pass
```

### ЁЯЯб MINOR тАФ `--since HH:MM` silently returns empty when time is in the future

**File:** `cli_server.py:186-200`

```python
h, m = str(since_hhmm).split(":")
now_dt = datetime.now()
since_dt = now_dt.replace(hour=int(h), minute=int(m), second=0, microsecond=0)
since_ts = since_dt.timestamp()
```

If the user types `--since 23:00` at 08:00, `since_dt` resolves to 23:00 **today** тАФ in the future. `since_ts > time.time()` means no done-events pass the `>= since_ts` filter and the output silently shows an empty event list. No error, no warning.

**Suggested fix:** After computing `since_dt`, check if it's in the future and if so subtract one day:
```python
if since_dt > now_dt:
    from datetime import timedelta
    since_dt -= timedelta(days=1)
since_ts = since_dt.timestamp()
```

## 2. Concurrency / Race

### тЬЕ No GIL race on `_last_send_ts`

`send()` and `_compute_last_progress_ts()` both run on the Qt main thread (QTcpServer callbacks are dispatched in the Qt event loop). The QTimer watchdog also fires on the main thread. There is **no concurrent access** тАФ Qt's event loop serialises all callbacks. CPython dict operations are additionally GIL-protected. No lock needed.

### тЬЕ IPC framing тАФ no partial-message race

`cli_server.py` uses `sock.canReadLine()` + `sock.readLine()` (Qt's line-buffered read). Each request is a single newline-terminated JSON object. Partial writes cannot produce a partial-parse because `readLine` only returns data when a `\n` is present in the buffer. тЬУ

### ЁЯЯб MINOR тАФ Inconsistent bytes vs str in `session.write()`

**Files:** `orchestrator.py` multiple locations

Most timer callbacks use `b"\r"` (bytes):
```python
lambda s=lead.session: s and s.write(b"\r")   # _flush_pending_lead_cc:1487
```
But `_auto_trust` uses `"\r"` (str):
```python
pane.session.write("\r")   # orchestrator.py:1294
```
And `_inject_idle_reminder` uses:
```python
pane.session.write(b"\r")  # orchestrator.py:1602
```

This inconsistency predates the current patch but new code (`_inject_auto_chain_handoff`, `_inject_idle_reminder`) follows the bytes convention. If `PtySession.write()` accepts both, no crash тАФ but the inconsistency is a latent bug waiting for a type-check pass to fail. **Not introduced by this patch; noting for completeness.**

## 3. Performance

### ЁЯЯб MINOR тАФ `pane_status_report` scans all session day-dirs on every call

**File:** `orchestrator.py:2107-2124`

```python
for day_dir in sorted(sessions_root.iterdir(), reverse=True):
    ...
    for f in sorted(proj_dir.iterdir()):
        if f.stat().st_mtime >= since_ts:
```

For each role, the code iterates ALL past day-directories (reverse-sorted) and stats every file to check `mtime >= since_ts`. With 30 days of sessions ├Ч 10 panes ├Ч N files per day, this is O(days ├Ч panes ├Ч files). Called only by `takkub status` (user-triggered), so not on the hot path. Acceptable now, but will degrade as session history grows.

**Optimisation opportunity:** Since `day_dir` is named `YYYY-MM-DD`, directories older than `since_ts` can be skipped without reading them:
```python
try:
    day_ts = datetime.strptime(day_dir.name, "%Y-%m-%d").timestamp()
    if day_ts + 86400 < since_ts:
        break  # all subsequent dirs are older
except ValueError:
    pass
```

### ЁЯЯб MINOR тАФ `pane_status_report` screenshot-finding sorts all files via stat

**File:** `orchestrator.py:2100-2102`

```python
shots = sorted(shot_dir.iterdir(), key=lambda f: f.stat().st_mtime, reverse=True)
if shots:
    last_screenshot = str(shots[0])
```

`sorted()` stats every file. With many screenshots, this is O(n├Чstat). Prefer `max()`:
```python
try:
    last_screenshot = str(max(shot_dir.iterdir(), key=lambda f: f.stat().st_mtime))
except (OSError, ValueError):
    pass
```
`ValueError` is raised by `max()` on an empty iterable.

### тЬЕ `takkub list` polling тАФ `list_status_detailed` complexity

O(panes ├Ч 2 stat calls) per `list_status_detailed` call. With ~10 panes = ~20 stat calls. Triggered by user `takkub list` only, not automatically. Acceptable.

## 4. Failure Modes

### тЬЕ Transcript path missing тЖТ graceful

`pane_status_report` guards with `if transcript_path:` before read. `_compute_last_progress_ts` guards with `if transcript_path:`. OSError on stat is caught. тЬУ

### тЬЕ Screenshot dir not created тЖТ graceful

`OSError` on `shot_dir.stat()` caught with `pass` in both `_compute_last_progress_ts` and `pane_status_report`. тЬУ

### тЬЕ Clock skew (mtime > now)

If `mtime > time.time()`, then `silent_for = now - last_ts` would be negative, which cannot exceed `STALL_THRESHOLD_SEC` (300). No stall flagged тАФ conservative, correct behavior. тЬУ

### ЁЯЯб MINOR тАФ Transcript tail contains raw ANSI escape codes

**File:** `orchestrator.py:2089-2093`

```python
raw = pathlib.Path(transcript_path).read_bytes()
lines = raw.decode("utf-8", errors="replace").splitlines()
tail_lines = [ln for ln in lines if ln.strip()][-5:]
```

PTY transcript contains ANSI control sequences (`\x1b[32m`, `\x1b[0m`, cursor moves). These appear verbatim in the `transcript_tail` output and in `_print_status_report`'s `line[:120]` truncation. The 120-char truncation may cut mid-escape, leaving dangling SGR sequences that bleed colour into the rest of the terminal output.

**Suggested fix:** Strip ANSI before display:
```python
import re
_ANSI = re.compile(r'\x1b\[[0-9;]*[mABCDHJKSThlsu]')

tail_lines_clean = [_ANSI.sub('', ln) for ln in tail_lines]
transcript_tail = "\n".join(tail_lines_clean)
```
Or apply at print time in `_print_status_report`.

### ЁЯЯб MINOR тАФ `pane_status_report` shows same screenshot for all roles

**File:** `orchestrator.py:2096-2103`

`last_screenshot` in each pane entry points to `runtime/exports/<date>/<project>/screenshots/<newest>` regardless of role. Backend and QA both show the same screenshot path. Misleading for the user тАФ the screenshot belongs to QA, not backend. 

Tied to the major finding in ┬з1 тАФ fix the scope of screenshot signal and this follows naturally.

### тЬЕ `_build_post_compact_brief` тАФ stale snapshot guard

`age > _POST_COMPACT_DETECT_SEC` (5 min) returns None. Teammate alive check before building brief. Hard cap at 2000 chars. тЬУ

## 5. Security

### ЁЯЯб MINOR тАФ Transcript tail in status output may contain sensitive content

**File:** `orchestrator.py:2089-2093`, `cli.py:197-200`

`takkub status` prints the last 5 lines of each pane's PTY output to the terminal. If a pane recently processed a file containing API keys, `git log` output with commit secrets, or `cat .env`, those lines appear in the status report. The operator themselves runs `takkub status`, so this is **acceptable exposure** (they already have access to those panes). Risk exists if the terminal session is recorded, shared via screen-share, or logged.

**No code change recommended** тАФ noting for operator awareness.

### тЬЕ `--since` injection

User-supplied `since` value goes through `str().split(":")` + `int()` only тАФ no shell execution, no SQL, no path construction. Safe. тЬУ

### тЬЕ `takkub status` command тАФ no Lead-token required

`status` is not in `_LEAD_ONLY_CMDS` тАФ any local process knowing the port can query pane status including transcript tails. The socket is loopback-only (LocalHost binding in `cli_server.py:50`), so no remote access. Single-user desktop app context тАФ **acceptable**.

### тЬЕ Double-gate on `_LEAD_ONLY_CMDS` (role field + token)

Layer 1 (from-field) + Layer 2 (HMAC-safe `secrets.compare_digest`) correctly guards `spawn/assign/close/close-all`. тЬУ

## Summary

| Severity | Count | Top Concerns |
|----------|-------|--------------|
| BLOCKER  | 0     | тАФ |
| MAJOR    | 1     | Screenshot dir mtime is project-scoped тЖТ suppresses stall alerts for non-QA roles when QA is active |
| MINOR    | 6     | `--since` future-time silent empty; ANSI codes in transcript tail; O(n) screenshot sort; session dir scan no early exit; misleading `last_screenshot` per-pane; `--since` AttributeError catch redundant |

**Top 3 concerns:**

1. **Screenshot signal scope (Major)** тАФ `_compute_last_progress_ts` uses `runtime/exports/<date>/<project>/screenshots/` for ALL roles. A QA screenshot suppresses stall detection for backend/frontend/devops in the same project. Fix: gate screenshot signal on QA/critic/designer roles only.

2. **`--since` future-time (Minor)** тАФ `takkub status --since 23:00` at 08:00 returns empty done-events silently. Subtract one day when `since_dt > now`.

3. **ANSI codes in transcript tail (Minor)** тАФ dangling escape sequences from PTY output can bleed colour into the user's terminal. Strip before display in `_print_status_report`.
