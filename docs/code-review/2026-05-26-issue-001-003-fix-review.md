## Fix B — `--since` Future-Time Wrap
**File:** `cli_server.py:195`  
**Verdict: PASS** (1 minor style issue)

### Logic correctness
```python
if since_dt > now_dt:
    from datetime import timedelta
    since_dt -= timedelta(days=1)
```

The logic is correct:

| Scenario | since_dt vs now_dt | Result |
|---|---|---|
| `--since 18:00` called at 20:00 | since_dt < now_dt | no wrap — correct |
| `--since 23:59` called at 10:00 | since_dt > now_dt | wrap to yesterday — correct |
| `--since 00:00` called at 00:00:01 | since_dt < now_dt | no wrap — correct |
| `--since 00:00` called at 23:59 | since_dt < now_dt | no wrap — correct |

The midnight edge case (`--since 00:00`) is handled correctly by the `>` comparison (not `>=`): today's 00:00 is always ≤ `now_dt`, so it never wraps.

The `except (ValueError, AttributeError)` catch-all is unaffected — the new branch is inside the `try` body, not restructuring the exception path.

**Test coverage:** `test_future_since_wraps_to_previous_day` in `TestSinceFutureTimeWraps` exercises the fix with `"23:59"` and asserts `since_ts <= time.time()`. ✅

### Minor style issue
`from datetime import timedelta` is inside the conditional branch instead of at the module or function top. Python caches imports so there's no correctness or performance issue, but intra-branch imports are a code smell and can surprise readers. Should be a top-of-file or top-of-function import.

## Fix C — ANSI Strip

### Layer choice

### Regex coverage — CSI sequences
The pattern `\x1b\[[0-9;]*[mABCDHJKSThlsu]` covers standard CSI final bytes:

| Sequence | Covered? |
|---|---|
| SGR colors/bold (`\x1b[32m`, `\x1b[0m`, `\x1b[1;33m`) | ✅ |
| Cursor movement (`\x1b[A`/`B`/`C`/`D`) | ✅ |
| Erase (`\x1b[J`, `\x1b[K`) | ✅ |
| Scroll (`\x1b[S`, `\x1b[T`) | ✅ |
| Cursor save/restore (`\x1b[s`, `\x1b[u`) | ✅ |

### **NEW FINDING: OSC and DEC private sequences not matched**

**DEC private parameters** (parameter byte `?`): sequences like `\x1b[?25l` (hide cursor), `\x1b[?2004h` (bracketed paste mode) use `?` as a parameter prefix. The pattern `[0-9;]*` does **not** include `?`, so these sequences are **not stripped** and will appear as literal garbage in `transcript_tail`.

Claude Code PTY output routinely emits bracketed paste enable/disable (`\x1b[?2004h`/`\x1b[?2004l`) around every prompt cycle.

**OSC sequences** (Operating System Command): `\x1b]...\x07` or `\x1b]...ST` are used for window titles and hyperlinks. These begin with `\x1b]` (not `\x1b[`) and are completely outside this regex. Modern terminal tools, including some Claude output formatters, emit OSC 8 hyperlinks. These would appear as binary garbage in `transcript_tail`.

**Suggested fix** — extend the regex to also strip DEC private and OSC:
```python
_ANSI = re.compile(
    r"\x1b\[[\x30-\x3f]*[\x20-\x2f]*[\x40-\x7e]"  # all CSI sequences (incl. DEC private)
    r"|\x1b\][^\x07\x1b]*(?:\x07|\x1b\\)"          # OSC sequences
    r"|\x1b[@-Z\\-_]"                               # two-byte ESC sequences (SS2, SS3, etc.)
)
```

Or more minimally, add `?` to the parameter class and add an OSC alternative:
```python
_ANSI = re.compile(
    r"\x1b\[[0-9;?]*[mABCDHJKSThlsu]"
    r"|\x1b\][^\x07]*\x07"
)
```

### Test coverage
- `test_transcript_tail_strips_ansi` tests SGR sequences (`\x1b[32m`, `\x1b[0m`, `\x1b[1;33m`) — passes with current regex ✅
- **No test** for DEC private sequences (`\x1b[?2004h`) or OSC sequences (`\x1b]0;title\x07`) — these would currently pass through unstripped
