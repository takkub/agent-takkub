2 · Data flow หลัก — คำสั่ง 1 อันเดินทางยังไง จบตรงไหน

ตามลูกศรจากบนลงล่าง: คุณพิมพ์ → Lead → assign → engine → pane ทำงาน → done → กลับ Lead → verify → ปิดงาน

1
คุณ (User)

พิมพ์สิ่งที่อยากได้ลงใน Lead pane — เช่น "ทำหน้า login + API"

2
Lead · routing

Lead อ่าน → routing_planner.classify() ตัดสินว่าใช้ role ไหน (frontend? backend? qa?) แบบ parallel หรือ sequential

ถ้าชัด → fire เลย · ถ้า ambiguous/irreversible → เสนอ propose table รอคุณ confirm ก่อน

3
Lead → CLI

Lead รัน takkub assign --role frontend --cwd <path> "task"

เขียนงานขึ้นต้น [ROLE: …] ปิดท้าย "รายงานด้วย takkub done"

4
CLI → cli_server → orchestrator

cli.py ส่งคำสั่งผ่าน TCP socket → cli_server → orchestrator (engine กลาง)

แยก process: CLI เบา ไม่ลาก engine 2.7k LOC เข้ามา (import-linter บังคับ)

5
orchestrator → spawn_engine

ถ้า pane ยังไม่เปิด → spawn_engine สร้าง PTY (ConPTY บน Windows / _pty_backend บน macOS) แล้ว spawn claude.exe

inject ให้ pane: role config (--mcp-config variant เฉพาะ role) + --plugin-dir (plugin ของ role) + env TAKKUB_ROLE/PROJECT/SHARD… + cwd + paste task + Enter

6
Teammate pane ทำงาน

claude.exe ตัวนั้นทำงานจริง (แก้โค้ด/เทส) เห็นสดในหน้าต่าง · คุยกับเพื่อนได้ takkub send --to backend (CC Lead อัตโนมัติ)

7
เสร็จ → takkub done

pane รัน takkub done "<สรุป>" → กลับผ่าน socket → orchestrator → lead_inbox

8
Lead รับ done → ตัดสินใจต่อ

[frontend done] … เด้งเข้า Lead → Lead ใช้ done-handoff rule:

DEV เสร็จหมด → verify sequence (มี docker? → devops ยก stack → QA ท้ายสุด) · ผ่าน → propose ship (commit/PR) · fail → propose fix loop

9
ปลายทาง (จบที่นี่)

done ทุกครั้ง → เขียน runtime/sessions/ + mirror ไป Obsidian vault + อัปเดต role-memory ของ role นั้น

งานจบ → Lead สรุป "ปิด session?" · takkub end-session เขียน summary · issue → GitHub

5 · คำสั่ง CLI (Lead/pane ใช้) + ปลายทางข้อมูล

⌨️ CLI หลัก (takkub …)

  • assign — spawn + ส่ง task ให้ role
  • send — peer message (CC Lead)
  • done — pane รายงานเสร็จ → Lead
  • list / status — สถานะ pane ทั้งหมด
  • goal — ตั้งเป้า session (prepend ทุก assign)
  • harvest — กู้งาน pane ที่ลืม done
  • doctor --fix — diagnose/ซ่อม env (claude/plugins/mcp)
  • services — docker compose ของ project
  • pipeline / issue / end-session / mcp / plugins

📤 ข้อมูลจบที่ไหน (sinks)

  • Lead inbox — done/report เด้งเข้า Lead ให้ตัดสินใจต่อ
  • runtime/sessions/ — log ราย role (per-machine, gitignored)
  • Obsidian vault — mirror decision notes (99-Logs / 01-Projects)
  • role-memory — ความรู้สะสมต่อ (role × project) ใช้รอบหน้า
  • GitHub — takkub issue → issue queue
  • git commit/PR — เฉพาะ Lead หลัง verify ผ่าน (คุณสั่ง)
สรุปทั้งระบบใน 1 ประโยค: คุณสั่ง Lead → Lead assign ผ่าน engine → engine spawn claude.exe pane ที่มี tool เฉพาะ role → pane ทำงานเสร็จ done กลับ Lead → Lead พา verify (QA ท้ายสุด) → เสนอ ship → log จบที่ sessions + vault + role-memory
