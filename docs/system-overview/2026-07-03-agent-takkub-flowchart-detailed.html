เริ่ม/จบ (สำเร็จ) terminal state process ◇ decision loop ย้อนกลับ

Phase A · Intent routing — routing_planner.classify()

ข้อความคุณถูกจับเข้า 1 ใน 7 kind (เช็คตามลำดับ บนลงล่าง — เจอก่อนชนะ)

keyword (ตัวอย่าง)primarycross-check
UI / page / form / button / CSSfrontend—
endpoint / API / schema / migrationbackend—
refactor / extract / migrate A→Bตามไฟล์+codex
design review / รีวิว UIcritic+gemini
UI e2e/smoke หลายหน้า (browser)qa --plan --shards—
feature ใหญ่ (UI+API)frontend + backend (parallel)—
complex / สงสัย approachprimary+gemini (1M ctx)

Phase B · Confirm & dispatch — propose → fire → fan-out

PROPOSE รอ user ตอบ · แล้ว exec-mode ตัดสินว่า 1 ตัว หรือ แตกขนาน

auto-chain: ใส่ --auto-chain บน impl assign → ทุก pane done ครบ → orchestrator inject handoff เข้า Lead อัตโนมัติ → fire verify sequence เอง (devops→qa) ไม่ต้อง propose

Phase C · Spawn pipeline — assign แล้วเกิดอะไรใน pane

orchestrator สร้าง pane + inject ทุกอย่างก่อน paste task (เจาะลึก argv + env จริง)

Phase D · Work & recovery — watchdog ทุก 5 วินาที

_check_idle_teammates วิ่งทุก tick ผ่าน gauntlet นี้ (เรียงตามลำดับ gate จริง)

delivery self-heal (#22/#26): หลัง paste ถ้า pane ยัง ready → เช็ค input box: มี content = Enter หาย → resend CR · ว่างเปล่า = paste หาย → re-paste payload ใหม่แล้วค่อย submit

Phase E · Done-handoff → verify → ship

done เด้งเข้า Lead → verify sequence (QA ท้ายสุดเสมอ) → propose ship

Phase F · ปลายทางข้อมูล (sinks) — จบที่ไหนบ้าง

อื่นๆ: takkub issue → GitHub issue queue · takkub end-session → session summary + vault mirror · commit/PR → git (Lead เท่านั้น หลัง verify)
ทั้งหมดใน 1 บรรทัด: classify → (propose/confirm) → dispatch (solo/parallel) → spawn+inject → work (watchdog คุ้มกัน) → done → verify (QA ท้ายสุด) → ship → log จบที่ sessions+vault+role-memory (แล้ววนใช้รอบหน้า)
