# #103 phase 3 — role .md template generation + doctor-installed defaults (backend#1)

Scope: `docs/plans/2026-07-09-core-upgrade-plan.md` item 6c — "default role+skill bundle as
package data, installed by `takkub doctor --fix`".

## Audit — what already exists (not touched)

Per the session goal's own lesson (phase 0 was already done and nearly re-implemented), audited
before writing anything:

- **Custom role creation** (`custom_roles.py`) — `create_role()` already validates, persists
  `~/.takkub/custom-roles.json`, and writes `<name>.md` under `CUSTOM_AGENTS_DIR`
  (`SETTINGS_HOME/agents`) with an atomic temp-file-then-rename sequence. `delete_role()` removes
  both the registry entry and the file. Fully working, including UI delete. **No changes made.**
- **New Role skill picker** (`settings_window.py` `_build_new_role_view` +
  `_new_role_skill_roots`/`_reload_new_role_skills`) — real `.claude/skills/*/SKILL.md` checkbox
  list backed by `skill_scan.scan_skills()`, embeds a "## Skills ที่เกี่ยวข้อง" block into the
  generated role instructions when checked. Fully working. **Only the root-resolution list was
  extended (see below) — the picker mechanism itself is untouched.**
- **Built-in role files as package data** — `.claude/agents/*.md` + `CLAUDE.md` were *already*
  shipped as package data before this task: `MANIFEST.in` includes them for the sdist path,
  `setup.py`'s custom `build_py` (`_stage_assets()`) copies them into
  `src/agent_takkub/_assets/` right before `pyproject.toml`'s `package-data` list is collected,
  and `config.AGENTS_DIR` (`ASSETS_ROOT/.claude/agents`) reads them directly at runtime — on an
  installed build `ASSETS_ROOT` resolves into the wheel's staged `_assets/`, on a dev checkout it
  resolves straight to the repo root. **This part of item 6c was already fully done.**
- **Role files self-heal on every spawn** — `config.agent_role_dir(role)` (called from
  `spawn_engine.py` on every pane spawn) unconditionally re-reads `AGENTS_DIR/<role>.md` (or the
  custom-role fallback) and rewrites `RUNTIME_DIR/agents/<role>/CLAUDE.md` fresh each time. There
  is no persistent, editable copy of a built-in role file that can go stale — it is regenerated
  from the package-data source on every single spawn. Consequently `doctor` cannot meaningfully
  "repair" a built-in role file at the runtime-staging layer: there is nothing to repair, the
  staging is always in sync with `AGENTS_DIR` by construction.
- **`doctor.check_installed_integrity()`** already existed with FAIL-severity checks for
  `assets-claude-md` and `assets-role-files` (missing/empty `ASSETS_ROOT` content on an installed
  build), both with a "reinstall" fix hint — no `auto_fix` callback, because `ASSETS_ROOT` on an
  installed build is read-only site-packages content baked into the wheel at build time; there is
  no on-disk source `doctor --fix` could copy from except a fresh package install. This is a
  deliberate, pre-existing pattern in this file (contrast with `check_mcps`/`check_env_path`,
  which *do* have `auto_fix` because their targets are genuinely writable/regenerable at runtime).

## What was actually missing

Only one concrete gap survived the audit: **the default skill bundle (`.claude/skills/*/SKILL.md`)
was never shipped as package data at all** — no `MANIFEST.in` entry, no `setup.py` staging step, no
`pyproject.toml` `package-data` glob, no `config.py` read-path constant. On an installed
(pip/npm) build, `ASSETS_ROOT/.claude/skills` simply didn't exist, and the New Role / Skill
Catalog pickers' only root (`config.REPO_ROOT`) resolves to an empty venv ancestor there (already
flagged as a known limitation in a prior session's learned notes) — so the 5 built-in skills
(`cockpit-ui-style`, `debug-mantra`, `management-talk`, `post-mortem`, `scrutinize`) were
completely invisible to those pickers on any installed cockpit.

## What was implemented

1. **`MANIFEST.in`** — added `recursive-include .claude/skills *.md` alongside the existing
   `.claude/agents` include, so an sdist carries the skill bundle too.
2. **`setup.py`** — extended `_stage_assets()` to copy `.claude/skills/<name>/SKILL.md` into
   `_assets/.claude/skills/<name>/SKILL.md`, mirroring the existing agents-staging step exactly.
   Deliberately **not** a hard build failure when `.claude/skills/` is missing/empty (unlike
   `CLAUDE.md`/agents, which do `raise RuntimeError`) — the skill bundle is supplementary
   reference material for the role-creation UI, not required for a pane to spawn, so a
   skills-less build is still a valid wheel. Only the real `<name>/SKILL.md` layout is staged
   (matches `skill_scan._skill_files`'s nested-glob half); a stray flat `*.md` directly under
   `.claude/skills/` is ignored by the staging step (it wouldn't be a real skill).
3. **`pyproject.toml`** — added `"_assets/.claude/skills/*/SKILL.md"` to the `package-data` glob
   list, same pattern as the agents entry.
4. **`config.py`** — added `SKILLS_DIR = ASSETS_ROOT / ".claude" / "skills"`, the read-path
   constant analogous to `AGENTS_DIR` (dev checkout → repo root; installed → staged wheel data).
5. **`doctor.py`** — added an `assets-skill-files` entry to `check_installed_integrity()`,
   parallel to the existing `assets-role-files` check. Deliberately **WARN, not FAIL** (unlike the
   CLAUDE.md/role-files checks) — a missing skill bundle degrades the New Role/Skill Catalog
   pickers to "no built-in skills shown," it does not break pane spawning the way a missing
   CLAUDE.md/role file would. No `auto_fix` callback, for the same reason `assets-role-files`
   doesn't have one: `ASSETS_ROOT` is read-only site-packages content on an installed build: the
   only real fix is a reinstall, which the fix-hint text says explicitly.
6. **`settings_window.py`** — extended `_new_role_skill_roots()` to also include
   `config.ASSETS_ROOT` alongside the existing `config.REPO_ROOT` fallback. This is what actually
   makes the newly-shipped bundle visible to the New Role / Skill Catalog pickers on an installed
   build (`REPO_ROOT` there is empty; `ASSETS_ROOT` is where the wheel staged the skills). On a
   dev checkout the two paths are identical, so this is a harmless duplicate root there.

## Explicit judgment call — what "doctor --fix ติดตั้ง/ซ่อม" does *not* do here

The task text asked for `doctor --fix` to "install/repair" role+skill files "for a new machine
that doesn't have the files, or files are missing/stale," with a caveat about not clobbering
user edits (hash-compare or backup). After tracing every real consumer of `AGENTS_DIR`/
`SKILLS_DIR` in this codebase, I could not find any *writable, live* location that built-in
role/skill content is ever copied into and could subsequently go stale or accumulate user edits —
`agent_role_dir()` regenerates role staging fresh on every spawn (self-healing, never stale), and
there is no existing mechanism that copies built-in skills anywhere at all (only project-local
`.claude/skills/`, which is the user's own project content, never touched by the cockpit). The
only failure mode that actually exists for `ASSETS_ROOT` content is "wheel installed without it,"
which the existing `check_installed_integrity()` pattern already correctly resolves via
"reinstall" (not `auto_fix`) for role files, and now the same pattern for skill files.

I judged that inventing a new writable install target (e.g. copying the bundle into
`~/.claude/skills/`, Claude Code's own global skills directory) was **out of scope and
behaviorally risky** without an explicit ask: it would make cockpit-dogfooding-only skills
(`cockpit-ui-style`, `debug-mantra`, etc.) appear as available skills in *every* Claude Code
project on the machine, not just cockpit's own repo or cockpit-spawned panes — a surprising,
global side effect for a "phase 3 role template" task. If Lead wants that behavior, it needs its
own explicit design decision (target directory, opt-in/out, collision handling with a
user-authored skill of the same name) rather than being folded silently into this fix.
