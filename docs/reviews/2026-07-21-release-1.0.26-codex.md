## ควรแก้ (7)

### S1 — Settings transaction rollback ไม่รวม model stores ใหม่

- `settings_window.py:697-703` snapshot provider config/pipeline/tools/skills แต่ไม่ snapshot `provider_models.path()` และ `role_models.path()`
- model ถูกเขียนก่อน stores อื่น (`settings_window.py:726-729`); ถ้า pipeline/tools save ล้มเหลว UI report failure แต่ model files ถูก persist บางส่วนแล้ว
- เพิ่มสอง path ใน transaction snapshot/rollback และ test failure หลัง model write

### S2 — Cursor discovery ควรรองรับ canonical alias และตรวจ identity

- spec หา `agent`/`agent.exe` อย่างเดียว (`provider_spec.py:531`); `agent` เป็นชื่อ generic และ code เองยอมรับ collision risk
- official docs ปัจจุบันใช้ `cursor-agent` เป็น canonical command ใน install/parameter examples แม้บาง marketing surface ยังมี `agent` alias
- รองรับทั้ง `cursor-agent` และ `agent`, prefer canonical name และ probe `--version`/signature ก่อนถือว่า available เพื่อไม่ spawn โปรแกรมอื่นที่ชื่อ agent

### S3 — Kimi installer ควรตาม command ที่ upstream รองรับจริง

- upstream แนะนำ `uv tool install --python 3.13 kimi-cli`; code ใช้ `uv tool install kimi-cli`
- ระบุ Python ที่ compatible ชัดเจน (Kimi รองรับ 3.12–3.14) และทดสอบ PATH refresh/post-install discovery บน Windows + macOS; ตอนนี้ package manager exit 0 แต่ shim ยังไม่อยู่ PATH จะถูกรายงาน failure แม้ install สำเร็จ

### S4 — model precedence tests ยังพิสูจน์ feature ไม่ครบ

- `tests/test_role_models.py` ทดสอบ storage เท่านั้น ไม่มี spawn assertion
- `tests/test_provider_models.py` ทดสอบ provider model และ env-empty แต่ไม่มี `role > provider` ทั้ง Claude/non-Claude, ไม่มี Lead, ไม่มี provider switch, ไม่มี project isolation และไม่มี unavailable-provider→Claude substitution
- เพิ่ม regression matrix เหล่านี้ก่อนแก้ M4/M5 เพื่อกันกลับมาอีก

### S5 — idle tests ดีขึ้นแต่ยัง white-box และชื่อบาง test เกินสิ่งที่ assert

- การแก้ 3 tests เดิมให้เรียก `_work_then_idle()` สอดคล้อง semantic latch ใหม่ ไม่ใช่แค่ลบ assertion ให้ผ่าน และ test fast-turn fallback ที่เพิ่มมาปิด poll-gap ได้จริง
- แต่ทุก test mock `shows_startup_marker`; ไม่มี test ของ detection จริงใน `PtySession`
- `test_queued_pane_that_never_ran_is_not_reminded` ตรวจแค่ ~46 วินาที ทั้งที่ fallback จะเตือนหลัง 180 วินาทีถ้าไม่มี startup marker; rename หรือ assert behavior ครบช่วงเวลา

### S6 — limit-status changes เป็น feature นอก release scope และ cross-process claim ยังไม่มี inter-process lock

- working tree เพิ่ม shared JSON cache/backoff, เปลี่ยน poll 120→600s และ nullable utilization จำนวนมาก แม้ scope ที่ประกาศมี 6 เรื่องอื่น
- fixed `.json.tmp` + load/merge/replace ไม่มี file lock; concurrent processes ที่เริ่มพร้อมกันยัง fetch พร้อมกันและเขียน tmp เดียวกันได้ ขณะที่ tests จำลองสอง stores แบบ sequential ใน process เดียว
- แนะนำ split ออกจาก 1.0.26 หรือเพิ่ม real multi-process contention test/locking ก่อนรวม ไม่ควรให้ provider release แบก telemetry behavior change ที่ไม่ได้ gate แยก

### S7 — docs/comments หลังถอด chips ยังชี้ให้เปิด provider ที่ status bar

- `CLAUDE.md:344`, `provider_config.py:214`, `lead_context.py:413`, `routing_planner.py:557` และ comments/docstrings อื่นยังบอกว่า provider toggle อยู่ status bar

`_PROVIDER_DESC` มีแค่ codex/gemini; OpenCode/Kimi/Cursor แสดง hint ว่าง ทั้งที่เป็น provider experimental ที่ยิ่งควรอธิบายข้อจำกัด

### N2 — docstrings หลายจุดยังระบุ provider แค่ codex/gemini

เช่น `provider_config.provider_for`, `_provider_available`, `effective_provider_for`, `pane_env.py`, `shared_dev_tools.py`; runtime ส่วนใหญ่ registry-driven แล้วแต่เอกสารทำให้ maintainer กลับไปสร้าง shortcut แบบเดิมได้

### N3 — `provider_install.py` ยังอ้าง “status-bar chip's Install action”

chip ถูกถอดแล้วและไม่มี install action ใน status bar; แก้ docstring/comment ให้ตรง surface จริง (`doctor`/CLI/Settings ถ้ามี)

### N4 — hardcoded model presets เป็น snapshot ที่ stale เร็วและบางค่าเป็น display label

Combo editable ช่วยลด impact แต่เลือก preset ที่ CLI ไม่รับจะทำให้ spawn fail. Prefer provider-native model listing/cache เมื่อทำได้ หรืออย่างน้อยแยก display label ออกจาก exact argv value และอัปเดต source/date ที่ verify

## คำตอบจุดที่ขอเน้น

### Model precedence

- Non-Claude teammate/Lead: `role > provider > CLI default` ถูกใน happy path
- Claude teammate: `TAKKUB_TEAMMATE_MODEL` explicit (รวม empty) > role > provider > tier; empty ยังแปลว่า **ไม่ส่ง `--model`** ถูกต้อง
- Claude Lead: **ผิด** — ไม่อ่าน role/provider model (M5)
- Provider switch/project/substitution: **ผิด** เพราะ role model ไม่ผูก provider/project (M4)

### `TAKKUB_TEAMMATE_MODEL=""`

implementation ที่ `spawn_engine.py:1651-1654` ถูกต้องและ regression test ผ่าน: ถ้า env key มีอยู่แต่ค่า empty จะไม่ fallback ไป role/provider/tier และไม่ append `--model`

### Idle latch

- short task ระหว่าง poll ticks ถูกแก้เพิ่มด้วย 180s fallback และ test ใหม่แล้ว
- regression เสี่ยงสุดยังอยู่ที่ stale/body startup marker ซึ่ง reset idle streak ก่อน fallback ทุกครั้ง (M3)

### Multi-provider / cross-platform

- registry-driven discovery/install/doctor และ `SUBPROCESS_NO_WINDOW=0` บน non-Windows เป็นทิศทางที่ดี
- claude-only shortcuts ที่ยังหลุด: task notice เฉพาะ Codex, ready wait เฉพาะ Codex/Gemini, Claude Lead model branch แยก behavior
- Kimi Windows env var ผิด และ Cursor upstream capabilities ถูก mark unconfirmed ทั้งที่ยืนยันได้แล้ว (M7)

### Test quality

- idle tests ที่แก้ 3 ตัว: semantic update มีเหตุผล แต่ยัง implementation-coupled; ไม่ใช่หลักฐานพอสำหรับ real screen marker
- `test_main_window_status_bar.py` ลบ 124 บรรทัด **เหมาะสม** เพราะ methods/widgets ถูกลบจริง; ไม่ควรเก็บ tests ของ dead UI. ถ้าต้องการ guard เพิ่ม ให้มี structural smoke assertion ว่า status bar ไม่มี provider chips และ Settings ยัง toggle ได้ (QA reports ปัจจุบันมี smoke evidence อยู่แล้ว)
- provider tests ที่ assert `ready_rules == ()`, `context_strategy == "none"`, `autonomy_flags == {}` เป็นการตรึง known gap ไม่ใช่ proof ว่า provider ใช้งานได้
