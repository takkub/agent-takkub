# Codex Review - agent-takkub (2026-05-20)

## TL;DR
The cockpit has a sensible architecture: a PyQt orchestrator owns pane state, exposes a loopback socket, and panes coordinate through the `takkub` CLI. The main risk is that several high-impact boundaries are still policy/prompt-level rather than enforced in the orchestrator/CLI.

Findings: 7 total (2 P0, 3 P1, 2 P2). The P0s are project-scoping/write-boundary bugs: arbitrary `--cwd` can place a pane outside its project while still stamped with that project, and Lead's `BLOCKED_DIRS` rule is prompt-only while panes run with permission bypass flags.

Recent commits improved prompt guidance and raised the CLI timeout from 5s to 15s, but the latest auto-routing behavior is mostly encoded in `CLAUDE.md`, not executable routing logic. That leaves edge cases dependent on Lead following instructions under context pressure.

## Findings

### [P0] `assign --cwd` is not constrained to the calling project
- **File:** src/agent_takkub/orchestrator.py:1137
- **Issue:** `assign()` forwards caller-provided `cwd` directly to `spawn()`, and `spawn()` prefers explicit `cwd` over `default_cwd_for_role(..., project=project_ns)`. The pane is still stamped with `TAKKUB_PROJECT=project_ns`, so a Lead in project A can accidentally spawn a teammate in project B's filesystem while all `send/list/done` events are recorded under project A.
- **Impact:** This is the most important project-scoping failure. It can produce wrong-repo edits, misleading audit trails, wrong Obsidian session notes, and follow-up messages routed to the wrong Lead namespace. The prompt says every row must have a cwd, but no code checks that the cwd belongs to `projects.json[project].paths`.
- **Fix:** Resolve and validate explicit cwd before spawning. Allow only paths under the selected project's configured paths, except for explicit cockpit/runtime exceptions if needed.

```python
def _allowed_project_roots(project: str) -> list[Path]:
    data = load_projects()
    proj = (data.get("projects") or {}).get(project) or {}
    return [Path(p).resolve() for p in (proj.get("paths") or {}).values()]

def _cwd_within_project(cwd: str, project: str) -> bool:
    target = Path(cwd).resolve()
    return any(target == root or root in target.parents for root in _allowed_project_roots(project))

# in spawn(), before choosing spawn_cwd from explicit cwd
if cwd and project_ns != "default" and not _cwd_within_project(cwd, project_ns):
    return False, f"cwd is outside project {project_ns}: {cwd}"
```

Add tests for `assign --cwd C:/other` from a `TAKKUB_PROJECT=pms` pane and for role-default cwd still working.

### [P0] Lead `BLOCKED_DIRS` is prompt-only while permission bypass is enabled
- **File:** src/agent_takkub/orchestrator.py:453
- **Issue:** `_render_lead_context()` injects a `BLOCKED_DIRS` paragraph telling Lead not to edit project paths, but enforcement stops there. The Claude pane is launched with `--dangerously-skip-permissions` at `src/agent_takkub/orchestrator.py:906`, and there is no PreToolUse hook, CLI-side gate, or filesystem sandbox preventing Lead from writing under those directories.
- **Impact:** The Lead direct-edit boundary is currently advisory. A compacted/confused Lead, MCP tool, plugin, or pasted task can still edit blocked project files directly, bypassing the intended specialist/audit workflow. This undermines the core "Lead plans, teammates edit" model.
- **Fix:** Move the boundary into an executable check. Options:

```python
# Spawn Lead with a cockpit-managed settings file containing hooks:
# - Edit/Write/MultiEdit/NotebookEdit deny when path is under BLOCKED_DIRS
# - Bash deny for shell redirections/copy/move/remove touching BLOCKED_DIRS if feasible
argv.extend(["--settings", str(render_lead_guard_settings(project_ns))])
```

If Claude hooks are too brittle, at minimum remove `--dangerously-skip-permissions` for Lead and add a generated local settings deny list for built-in write tools. Treat the prompt as UX, not security.

### [P1] Lead context can inject the wrong project's `BLOCKED_DIRS`
- **File:** src/agent_takkub/orchestrator.py:470
- **Issue:** `spawn(role_name="lead", project=project_ns)` correctly chooses `lead_cwd(project=project_ns)`, but `_render_lead_context()` ignores `project_ns` and calls `active_project()`. In multi-tab/background-tab flows, active project and target project can diverge.
- **Impact:** A Lead for project A can spawn with project A cwd but receive project B's blocked directories. That creates both false negatives (A paths not blocked) and false positives (B paths blocked in A's prompt), and it makes the direct-edit policy unreliable exactly when multi-project isolation matters.
- **Fix:** Pass `project_ns` into `_render_lead_context(project)` and resolve paths through the same project-aware helper used by `lead_cwd()` and `default_cwd_for_role()`.

```python
def _render_lead_context(project: str | None = None) -> str | None:
    data = load_projects()
    name = project or data.get("active")
    proj = (data.get("projects") or {}).get(name, {})
    paths = list((proj.get("paths") or {}).values())
```

Then call `role_md_file = _render_lead_context(project_ns)` in `spawn()`.

### [P1] Recent-exit resume state is keyed only by role, not by project
- **File:** src/agent_takkub/orchestrator.py:594
- **Issue:** `_recent_exits` is `role -> {cwd, ts}`. Exit, spawn, Codex/Gemini spawn, and `restore_teammates()` all read/write/delete by role only. Other state maps were correctly moved to `<project>::<role>`, but this one was not.
- **Impact:** In multi-tab usage, a `backend` exit in project A can overwrite or clear the `backend` resume record for project B. If both projects share a cwd basename or a common monorepo root, this can trigger the wrong `--continue` decision or suppress an intended resume. It also makes auto-respawn behavior harder to reason about when two projects have the same role open.
- **Fix:** Key `_recent_exits` by `(project, role)` or string key `f"{project}::{role}"`, and update all accesses at lines 755, 804, 1019, 1042, 1070, and 1517. Extend `tests/test_session_resume.py` to cover two projects with the same role.

### [P1] Peer-to-peer CC can silently drop the Lead audit copy
- **File:** src/agent_takkub/orchestrator.py:1273
- **Issue:** `send()` writes the primary peer message first, then best-effort CCs Lead only if the Lead pane exists and is alive. If Lead is not running, still booting, crashed, or not ready, the peer message succeeds without durable audit. There is no queued CC, event-level full body, or user-visible warning; `_log_event()` stores only `msg_preview`.
- **Impact:** The documentation promises peer messages are auto-CC'd to Lead, but the code can lose the audit trail. This matters for security and coordination: a frontend/backend exchange can change implementation direction without Lead seeing it, and later `done` summaries may omit why.
- **Fix:** Treat Lead CC as part of the operation's contract. Either return a warning/failure when CC cannot be delivered, or persist full peer-message events to a project-scoped audit log and replay missed CCs when Lead resumes.

```python
if should_cc and not lead_alive:
    _log_event("send_cc_missed", project=project_ns, to=to_role, from_=from_role)
    return False, f"sent to {to_role}, but Lead CC was not delivered"
```

### [P2] Auto-routing behavior is not testable because it lives in prompt/docs
- **File:** CLAUDE.md:207
- **Issue:** The recent commits `feat(lead): auto-routing propose-then-fire behavior` and `feat(lead): hybrid auto-fire for reversible tasks` updated `CLAUDE.md`, while only the socket timeout changed in Python. There is no parser/state machine representing "propose", "confirmed", "edit + fire", "ambiguous ok", or "auto-fire one-shot only".
- **Impact:** Edge cases depend entirely on Lead following a long prompt: "ok but...", mixed informational/actionable prompts, a teammate `done` that should propose verification but not chain fire, or a role-specific request that skips proposal. These are exactly the cases most likely to regress under context compaction.
- **Fix:** Introduce a small routing planner module that classifies user/Lead commands and emits structured actions (`PROPOSE`, `FIRE_ASSIGN`, `FIRE_ONESHOT`, `ASK_CLARIFY`). Keep Lead's prompt as explanation, but add unit tests for the documented examples and Thai/English confirmation variants.

### [P2] MCP write tools are pre-allowed for every Claude pane
- **File:** src/agent_takkub/shared_dev_tools.py:104
- **Issue:** The shared MCP config pre-allows PMS read and write tools (`pms_create_task`, `pms_update_task`, `pms_add_comment`) and `spawn()` injects the same shared MCP config into every Claude pane. There is no role split between Lead and teammates, and no per-project scoping of MCP permissions.
- **Impact:** This is convenient, but it widens the blast radius. A teammate pane or a prompt-injected MCP response can mutate PMS state without Lead confirmation. It also weakens the intended "Lead owns orchestration and external coordination" boundary.
- **Fix:** Generate role-specific MCP configs: Lead gets write tools, teammates get read-only tools unless explicitly enabled. Keep browser MCPs shared, but split PMS permissions by role.

## Architecture notes
The current design is straightforward and mostly coherent:

- Desktop app: PyQt6 `MainWindow` owns tabs and visual pane layout. Each project tab has one Lead and a teammate stack.
- Orchestrator: `Orchestrator` owns pane state, spawn/assign/send/close/done operations, auto-respawn, idle reminders, hot.md/session mirrors, and project namespace routing.
- CLI server: `CliServer` listens on loopback TCP and dispatches newline-delimited JSON to the orchestrator on the Qt thread, which serializes state mutations naturally.
- CLI: `takkub` reads `runtime/port`, stamps requests with `TAKKUB_ROLE` and `TAKKUB_PROJECT`, and enforces the teammate-vs-Lead command gate for spawn/assign/close.
- Panes: Claude/Codex/Gemini are spawned as PTY sessions with injected env vars. Claude panes get cockpit-managed prompt files, plugin dirs, MCP config, and permission/tool flags.

The project namespace work is directionally right: `TAKKUB_PROJECT` is stamped in pane env, `CliServer` passes `from_project`, and most orchestrator maps now use project-aware lookup. The weak points are the remaining active-project fallbacks and state maps that are still keyed only by role.

The timeout fix from 5s to 15s in `cli.py` is reasonable for slow pane spawns, but it does not solve the deeper issue that `assign()` returns before `_send_when_ready()` has delivered the task. A future enhancement should expose a queued/delivered status event so Lead can distinguish "socket accepted" from "task actually pasted".

## Suggested next actions
1. Add cwd validation in `spawn()`/`assign()` and tests that prove explicit cwd cannot leave the caller's project.
2. Replace prompt-only `BLOCKED_DIRS` with an executable write guard for Lead, or remove Lead permission bypass until a guard exists.
3. Change `_render_lead_context(project_ns)` and `_recent_exits[f"{project}::{role}"]` to close the remaining multi-tab scoping holes.
4. Make peer-to-peer Lead CC durable or fail-visible.
5. Extract auto-routing policy from `CLAUDE.md` into a small tested planner, keeping prompt text as guidance only.
6. Split shared MCP config by role so write-capable external tools are not pre-allowed in every pane.
