### Mantra step 2 — fail path (as documented at the time of the original fix)

The root cause is written up in full in
`docs/reviews/2026-07-10-resume-bridge-race-FIX.md`: a paint-gap race —
`SessionStart(source="resume")` could start polling
`is_at_ready_prompt()` for `/remote-control` injection in the brief window
before claude's picker had actually repainted over the previous idle footer,
so the poll misread the pane as ready, injected `/remote-control` + a
delayed auto-Enter, and that Enter landed on the picker instead — observed
as "Resume cancelled".

That doc's own fix (deferring the `resume`/`compact` sources to the idle-tick
reap path, plus an explicit `is_at_resume_picker()` pre-check) was layered
on top of *prior* attempts (#107/#110/#112) that had already tried to close
this race and lost to it repeatedly over two days of live reports. Per
commit `28136df`'s message, that layered fix was itself still not durable
enough ("ทุกfix แพ้ paint-gap" — every fix lost to the paint gap), so the
final resolution was to remove the auto-injection path entirely rather than
add a fourth guard on top of three: no code path types `/resume` or
`/remote-control` into the Lead pane automatically anymore. The user types
both directly.

### Mantra step 3 — falsify

Hypothesis: "the picker-race can still happen because some other call site
still auto-injects a slash command into the Lead pane on `/resume`."
Disproof attempted: grepped every caller of `inject_slash_command_when_ready`
across `src/` — the only call site left is the internal
queue-continuation self-call inside the function itself
(`lead_inbox.py:395`, draining `_slash_inject_queue` for other still-active
callers). Zero external callers. Hypothesis falsified — nothing fires this
path anymore for `/resume`.

Checked the one remaining resume mechanism — the mobile PWA resume endpoint
(`remote/api.py:239` `resume_lead`) — to confirm it doesn't share the race.
It doesn't: it closes the Lead pane (`orch.close(..., force=True)`) and
respawns it with the native `--resume <session_uuid>` CLI flag
(`orch.spawn(..., resume_uuid=session_uuid)`), which claude resolves
non-interactively at process start — no TUI picker is ever shown, so there
is no picker to race against.

## #114 — bare-Esc + Ctrl+W draft_len drift

**Already fixed by commit `4f07caf`** (2026-07-10 19:34,
`fix(lead): #114 A3-secondary — Ctrl+W/Alt-Backspace draft_len drift`).

Verified `src/agent_takkub/lead_draft_state.py` against the task's two asks:

1. **Ctrl+W (0x17) / Alt+Backspace (`\x1b\x7f` / `\x1b\x08`) tracking** —
   present. `LeadDraftState.word_len` tracks chars typed since the last
   whitespace in the current NONEMPTY run; `_apply_word_delete()` decrements
   `draft_len` by the real word length in the common case (exact), falling
   back to a conservative 1-char step once `word_len` is exhausted (biased
   toward overcounting — safe, since overcounting only delays injection past
   `DRAFT_HOLD_TIMEOUT_S` rather than letting an injection clobber real text).
   Both `_CTRL_W` (bare 0x17) and the Esc+Backspace/Del encoding of
   Alt+Backspace are wired in `advance_draft_state()`.

2. **Bare-Esc vs Esc-closes-menu split** — deliberately **not** attempted, by
   design, documented at the handling site (`lead_draft_state.py:256-275`)
   and in full in `docs/reviews/2026-07-10-a3-secondary-esc-analysis.md`:
   this module only observes the byte stream going *into* the pane
   (`advance_draft_state()` is fed exclusively from
   `Orchestrator._on_pane_input`, confirmed by reading `lead_inbox.py` — no
   render/output-side signal reaches it) and a lone `0x1B` is genuinely
   indistinguishable at that layer between "user is clearing an in-progress
   draft" and "user is dismissing claude's own slash-menu/dialog". The
   existing behavior (always clear on bare Esc) is kept as the safer of the
   two wrong-answer directions: over-clearing risks an injection landing on
   an empty line a beat early; under-clearing risks `draft_len` sticking
   "nonempty" over text that no longer exists, silently blocking injection
   until the 180s hold timeout spills it. A real fix would need render-side
   menu detection (the same footer-text approach `pty_session.is_at_resume_picker()`
   uses for #113) — out of this pure byte-level module's scope, and not
   requested as new architecture by this task.
