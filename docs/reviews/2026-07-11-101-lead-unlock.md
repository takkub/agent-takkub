# #101 — Unlock Lead provider (degraded mode, M-L)

Audit + implementation notes for backend#2 (Wave 3). Scope: unlock Lead so
it can be backed by `codex`/`gemini` (agy), not just `claude` — degraded
mode first (full parity, a transcript adapter + per-CLI resume, is L-XL and
explicitly out of scope here).

## Audit: was any of this already done?

Confirmed via `git log`/code read at the start of this task: nothing in
this repo touched `_FORCED_PROVIDER["lead"]` or built a non-claude Lead
context path before this change. Issue #103 Phase 0 (`provider_spec.py`)
already laid the groundwork this task consumes — the `ProviderSpec`
dataclass's `supports_mirror` / `supports_resume` / `supports_token_meter`
/ `supports_remote_history` / `supports_hooks` / `produces_jsonl_transcript`
fields existed but were **unwired** (no call site read them) before this
change. Nothing was duplicated.

## The 6 claude-specific coupling points (from the issue)

| # | Coupling point | Where | Fix in this change |
|---|---|---|---|
| 1 | CLAUDE.md auto-discovery + `--append-system-prompt-file` | `lead_context.py`, `spawn_engine.py` claude branch | New `render_lead_agents_md()` writes the SAME content to `AGENTS.md` (codex/agy's auto-discovery file) for a non-claude Lead |
| 2 | `--continue`/`--resume` | `spawn_engine.py` claude branch, `remote/api.py resume_lead` | Codex/gemini spawn branches never accepted `resume_uuid` to begin with (silently ignored pre-#101 too — teammates never had resume). For Lead specifically, `remote/api.py`'s `resume_lead` now gates on `provider_config.lead_missing_capability("supports_resume")` and returns a **409 with a clear message** instead of reaching `spawn()` and failing opaquely |
| 3 | Mobile mirror reads claude JSONL | `remote/notify.py` (`resolve_lead_jsonl`, `list_recent_lead_sessions`) | Already gracefully returned "nothing" when no session_uuid/jsonl exists (a non-claude Lead never gets a `session_uuid` stamped — no code path sets it). What was missing: a WHY. `remote/api.py`'s `lead_history`/`lead_sessions` now include a `lead_provider_note` field explaining the gap |
| 4 | `/remote-control` slash / PWA control | `remote/api.py` | Same code paths as #2/#3 above — the remote-control server's Lead-facing endpoints (`lead_history`, `lead_sessions`, `resume_lead`) are the full surface; no separate claude slash-command wiring exists in this codebase (remote-control is the cockpit's own HTTP+SSE server, not a Claude Code slash command) |
| 5 | `SessionStart` hook (`takkub session-report`) | `.claude/settings.json` hooks, read only by claude itself | Inherent, not a code gate: codex/agy have no hooks mechanism at all, so there is nothing to wire or fail — the hook config simply never runs for a non-claude Lead. `session_uuid` on `pane_state` stays unset for that pane, which is exactly the signal #3's gate reads |
| 6 | Token/limit meter (JSONL-based) | `limit_panel.py` / `limit_status.py` | Softer gap than the other 5: `LimitStore` polls the account's Claude config dir (`user_profile.config_dir_for`), reflecting overall Claude Max/Pro usage — not specifically "Lead's own spend". It keeps working (still useful — teammates can be claude-backed even when Lead isn't) but no longer reflects Lead's own token consumption once Lead is on codex/agy. Documented via the `supports_token_meter` label in `lead_capability_gap()`; no separate code gate needed since nothing *breaks*, it just stops being about Lead specifically |

## What changed

- **`provider_config.py`**: `lead` removed from `_FORCED_PROVIDER` — it now
  resolves through the normal `role-providers.json` override path, default
  `claude` unchanged. Two new helpers:
  - `lead_capability_gap(project) -> (provider, [missing labels]) | None` —
    full gap report, used by the Settings UI badge and the mobile
    history/sessions endpoints.
  - `lead_missing_capability(flag, project) -> provider | None` — precise
    single-capability check, used by `resume_lead` (gates on
    `"supports_resume"` specifically, not "any gap", so a future provider
    with partial capability isn't wrongly blocked/allowed).
- **`lead_context.py`**: extracted `_build_lead_context_text()` (the
  cockpit CLAUDE.md + BLOCKED_DIRS + substitution/parallel/session-brief
  content) out of `_render_lead_context()`, which still writes it to
  `runtime/lead-context.md` for claude's `--append-system-prompt-file`.
  New `render_lead_agents_md(project, spawn_cwd)` writes the SAME text to
  `<spawn_cwd>/AGENTS.md`, marked with `codex_agents_md.TAKKUB_MARKER` so a
  user-owned AGENTS.md is never clobbered — deliberately bypassing
  `ensure_agents_md()`'s generic teammate cheatsheet (that text says "you
  are a specialist, do the task yourself, call takkub done", which is
  backwards for the orchestrator role).
- **`spawn_engine.py`**: the GEMINI/CODEX branches (already shared by every
  teammate role) gained an `_is_lead` special case:
  - cwd resolves via `lead_cwd(project)` (project root), not
    `default_cwd_for_role()` (a teammate staging dir).
  - context is planted via `render_lead_agents_md()`, not
    `ensure_agents_md()`.
  - env uses `_build_lead_env()` (git/gh passthrough) and
    `TAKKUB_LEAD_TOKEN` (Lead-only takkub CLI auth) instead of
    `_build_pane_env()` + a per-pane teammate token — this was the
    sharpest bug risk: without it, a codex/gemini-backed Lead would mint a
    **teammate** token and every `takkub assign/close/broadcast` call from
    that pane would be rejected by `cli_server.py`'s Lead-only auth check,
    silently crippling orchestration.
  - The pre-existing claude branch (unchanged) still handles the
    `effective_provider == CLAUDE` case, including the automatic
    "unavailable codex/gemini degrades to claude" substitution — a Lead
    pointed at a toggled-off or uninstalled provider falls through to
    claude exactly like any other role, so `lead_capability_gap()` reports
    `None` (no gap) in that case, matching reality.
- **`settings_window.py`**: the previously hardcoded, always-locked "Lead:
  Claude (fixed)" row in Providers & Roles is now a real CLI combo (via
  `_overridable_roles()` prepending `"lead"` when it's not in
  `FORCED_ROLES`), with `show_enable_toggle=False` (Lead isn't a
  dev-pipeline participant — no enable/disable toggle makes sense for it)
  and a dedicated `⚠ non-Claude` capability-warning badge that appears
  whenever the combo's current selection isn't claude, listing what's lost.
- **`cockpit_theme.py`**: `#capabilityWarning` QLabel style (same amber
  palette as the existing `#substituteBadge`, bolder weight).
- **`remote/api.py`**: `lead_history`/`lead_sessions` responses gained a
  `lead_provider_note` field (`None` when Lead is claude); `resume_lead`
  gates explicitly on `supports_resume` before doing any uuid/cwd work.

## What did NOT change (scope boundary honored)

- `provider_spec.py` — zero edits. All capability flags this change reads
  (`supports_mirror`, `supports_resume`, `supports_remote_history`,
  `supports_token_meter`, `supports_hooks`) already existed from #103
  Phase 0; this change is purely a consumer.
- MCP-injection zone in `spawn_engine.py` (backend#1's #100 territory) —
  untouched; the `_is_lead` branches added here sit entirely within the
  pre-existing GEMINI/CODEX cwd/context/env blocks, before any MCP-config
  wiring.
- Claude Lead behavior — the claude branch of `spawn()`, `_render_lead_context`
  (still writes `runtime/lead-context.md` exactly as before), and every
  existing claude-Lead test pass unmodified.

## Default behavior

Still `claude`. Unlock is opt-in via `role-providers.json` (hand-edit) or
the Settings → Providers & Roles → Lead row combo. No behavior changes for
any user who hasn't explicitly remapped `lead`.

## Known remaining gaps (not fixed here — flagged, not silent)

- `resume_lead`'s early-gate covers the mobile PWA's Resume button. The
  desktop cockpit's own respawn/crash-recovery path
  (`_toctou_redefer`/`_retry_deferred_spawn`) can still pass a
  `resume_uuid` through to a codex/gemini-backed Lead's spawn call — it's
  simply ignored (never reaches a `--resume`-shaped flag in either
  branch), which is silent-degrade-safe (no crash, no wrong behavior) but
  not loudly announced to the Lead itself. Full parity here needs a
  per-provider resume story (L-XL, out of scope).
- Token/limit meter (#6 above): no dedicated "Lead's own usage" concept
  exists even for claude-backed Lead today (it's account-wide), so there
  was no narrower behavior to preserve or gate — documented as a soft gap,
  not fixed.
