# Audit: Queue / fan-out cap enforcement

**Date:** 2026-06-30 · **Author:** Lead (Claude) · **Scope:** read-only audit, no code change
**Question:** Does anything enforce a cap on the number of concurrently-active panes per `machine_fanout_cap()`, or is fan-out purely Lead-self-limited? I.e. is the red-team's "Queue-based Wave Executor" already partly built?

---

## TL;DR — there is **NO central cap enforcement**. `machine_fanout_cap()` is **advisory text** injected into the Lead prompt; the Lead self-limits. The Queue Executor is the **one genuinely net-new** item of the three backlog features (Reaper = done, Webhook = new, Queue = new **and the real gap**).

---

## What actually exists

| Mechanism | What it does | What it does **not** do |
|---|---|---|
| `exec_mode.machine_fanout_cap()` (CPU/RAM-derived) | Builds an **advisory string** injected into the Lead's context (`orchestrator.py:1013`): *"Cap K at {cap} for this machine — split bigger batches into waves."* | Does **not** gate spawning. Nothing rejects/queues an assign that exceeds the cap. |
| `PaneRegistry.spawn_queue` (FIFO, `spawn_engine.py:227`) + `spawn_in_progress` | Serialises **ConPTY construction** so only one `session.spawn()` runs at a time (avoids native spawn races). | Does **not** limit how many panes end up **active**. It throttles the *rate* of construction, not the *count* of live panes. |
| `spawn_deferred` set | Dedups duplicate QTimer spawn callbacks for one (project::role). | Not a concurrency cap. |

**So:** if the Lead fires 6 `takkub assign` calls, all 6 panes spawn (serialised one-construction-at-a-time, then all live concurrently). The only thing keeping fan-out within machine limits today is **the Lead obeying the advisory cap in its prompt**. If the Lead miscounts — or a user manually fires many assigns — nothing stops the machine from being oversubscribed (the exact RAM-blowout the red-team worried about).

This matches the red-team's own Feature-G critique: assigns arrive as **independent CLI/IPC calls** with no pre-aggregated plan, so there's no natural choke point — except the spawn path itself, which is where a real queue would live.

## Designing the Queue Executor (if/when built — Phase 3)

The serialised spawn path is the right insertion point:

1. **Gate at spawn:** when an assign arrives and `count(active non-lead panes in project) >= machine_fanout_cap()`, **don't spawn** — enqueue the request (role + task spec + cwd + auto_chain/requires_commit/shard flags) into a **pending-assign queue** (distinct from the construction `spawn_queue`).
2. **Dequeue on completion:** in `done()`/`close()`, when a pane frees a slot, pop the next pending assign and spawn it. Re-evaluate `machine_fanout_cap()` at dequeue time (it's dynamic / RAM-sensitive).
3. **Surface state:** queued assigns must show in `takkub list` / `takkub status` as `queued (waiting for slot)` so the Lead and user aren't confused by a pane that "didn't appear." Emit a `[system]` notice on enqueue.
4. **Durability:** persist the pending queue (like `_pending_done_notices`) so a cockpit restart doesn't drop queued work.

**Effort:** Medium. Touches `spawn_engine` (active-count gate + pending queue), `done()`/`close()` (dequeue trigger), `list_status` (surface queued state), and a durability file. **Risk:** changes spawn UX (a pane may not appear immediately) — needs a clear "queued" signal or it reads as a bug.

## Recommendation

1. **This is the one Phase-3 item worth building** — it's the only backlog feature that isn't already shipped, and it directly serves the "don't have to babysit fan-out / don't crash the machine" goal.
2. **Do NOT build it unattended.** It changes live orchestration UX and needs design buy-in (queued-state surfacing, cap re-evaluation policy, durability). Build it with the operator awake to watch the dogfood — which is also exactly where the multi-dev system would be a fair test, *after* the design is agreed.
3. **Cheaper interim mitigation (optional, ~10 lines):** a soft guard that, when a spawn would exceed `machine_fanout_cap()`, emits a `[system]` **warning to Lead** ("over fan-out cap — consider waves") without blocking. Catches Lead miscounts without the full queue's UX change. Could ship before the full executor.

## Verdict for the feature backlog

Of the three red-team features: **Reaper = already done**, **Webhook = net-new (self-contained, low risk)**, **Queue Executor = net-new and the real architectural gap**. Recommended order when the operator is awake: Webhook first (isolated, low risk, high "know when I'm away" value), then design + build the Queue Executor with oversight. The optional soft-cap warning can land anytime as a cheap stopgap.
