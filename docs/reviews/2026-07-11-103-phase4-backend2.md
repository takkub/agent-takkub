# #103 phase 4: skill library + per-provider skill injection

Wave 3 note (`docs/plans/2026-07-09-core-upgrade-plan.md` §6, phase 4):
"skill library + matrix (skill semantics differ per provider — resolved per
ProviderSpec from phase 0)". This is that phase.

## Audit — what already existed before this task

| Piece | File | What it does |
|---|---|---|
| Skill scanner | `skill_scan.py` | `scan_skills(roots)` — reads real `.claude/skills/*/SKILL.md` frontmatter (name/description), dedup first-root-wins. Already used by two callers. |
| Role-scope audit | `skill_audit.py` | TF-IDF cosine overlap between **role instruction docs** — a different concern (role territory, not skill files). Backs "Role Overlap" view. |
| Skill Catalog view | `settings_window.py` (`VIEW_SKILL_CATALOG`, index 8) | Read-only browser: lists real skills via `skill_scan`, shows which role docs mention each one (substring match). No policy, no spawn-time effect. |
| New Role skill-picker | `settings_window.py` (`_build_new_role_view`, `_append_skill_references`) | Checkboxes at role-**creation** time. Checked skills get baked as a `"## Skills ที่เกี่ยวข้อง"` block into the CUSTOM role's saved `.md` **once**, at create time. Only applies to brand-new custom roles — no way to change a BUILT-IN role's (frontend/backend/qa/...) skill set, and no way to change it later without re-editing the role file by hand. |
| Per-provider context strategy | `provider_spec.py` (`ProviderSpec.context_strategy`) | Field already existed (`"append_system_prompt_file"` for claude, `"agents_md_file"` for codex/gemini) but was **undocumented/unwired** — grepping the whole tree found zero call sites reading it before this task. |
| AGENTS.md bridge | `codex_agents_md.py` (`ensure_agents_md`) | Plants one shared, provider-generic cheatsheet into codex/gemini panes' cwd. No per-role content — same file regardless of which role spawned it. |

**Gap confirmed**: nothing resembling the MCP/Plugins Matrix existed for skills.
`pane_tools_policy.py`'s pattern (role → allowlist, resolved fresh per spawn,
`~/.takkub/pane-tools.json`) had no skill equivalent — the only "skill → role"
wiring was the one-shot New-Role-creation embed above, which is a **different
mechanism** (bakes into the role file permanently, custom roles only, no
spawn-time resolution, no UI to change an existing role's set).

Scope of the task's item 2 (inject at spawn per provider) turned out **not**
too large once the matrix (item 1) existed — both were completed. No fallback
to "matrix only + documented gap" was needed for the happy path; the
architectural gap that **is** real and documented is the shared-AGENTS.md-file
race (see "Known gap" below).

## What was built

### 1. `skill_policy.py` (new) — per-role skill policy

Mirrors `pane_tools_policy.py`'s exact shape: `~/.takkub/skill-policy.json`,
atomic tmp+replace, `load_policy`/`save_policy`/`effective_skills`/
`set_role_skills`. Deliberately a **separate file/module**, not folded into
`pane_tools_policy.py` — different concern (context injection content, not
MCP/plugin pane access) even though the on-disk shape rhymes; the task spec
explicitly allowed either "pane-tools.json หรือไฟล์ใหม่ข้างๆ".

`skill_matrix_roles()` intentionally does **not** reuse
`pane_tools_dialog.matrix_roles()` — that helper excludes `codex`/`gemini`
because MCP config genuinely doesn't apply to them. Skills are the opposite:
codex/gemini are half the point of this policy (the AGENTS.md bridge). Only
`shell` (a plain terminal, no agent) is excluded.

`render_skill_appendix(role, roots, context_strategy)` is the spawn-time
renderer, shared by all three provider branches in `spawn_engine.py`:

- Filters the role's assigned skill names down to skills that actually
  **exist** under `roots` (a policy entry naming a skill this project lacks
  is silently dropped — same tolerance as every other policy loader here).
- `context_strategy == "append_system_prompt_file"` (claude): short nudge
  block ("อ่าน skill: X — description") — claude's own Skill tool already
  auto-discovers `.claude/skills/` regardless of this policy; this just tells
  the pane which ones matter for ITS role instead of leaving it to stumble
  onto them.
- `context_strategy == "agents_md_file"` (codex/gemini): a heavier block that
  names each skill, its description, and its **file path**, plus an explicit
  disclaimer: "⚠️ CLI นี้ไม่มีระบบ Skill tool ... ข้ามผ่าน bridge นี้ไม่ได้" —
  the agent has to `Read()` the file itself; any skill capability that
  depends on the Skill tool's own execution machinery (auto-trigger,
  bundled scripts the tool runs on the agent's behalf) is explicitly flagged
  as NOT bridgeable this way. This is the documented gap the task asked for.
- Any other `context_strategy` value → `""` (no known bridge).

### 2. Settings UI — "Skill Matrix" view (SKILL section, index 9)

New `VIEW_SKILL_MATRIX` alongside the existing "Skill Catalog". Same
role×item `QGridLayout` toggle grid as MCP Matrix / Plugins Matrix — reused
`_populate_matrix_grid` and `pane_tools_dialog.build_matrix` /
`matrix_to_role_items` / `diff_role_items` **verbatim** (those helpers were
already item-agnostic; no changes needed there). Wired into `_on_reset_clicked`
and `_on_save_apply_clicked` (own snapshot/rollback path via
`skill_policy.SKILL_POLICY_FILE`, alongside the existing MCP/Plugins/pipeline
snapshots).

The New Role picker's one-shot embed (`_append_skill_references`) is
**unchanged** — it still exists for the "bake a starter reference into a
brand-new custom role's instructions" use case. The Skill Matrix is the
generalization: works for every role (built-in + custom), resolved fresh on
every spawn, changeable any time without touching a role's `.md` file.

### 3. `spawn_engine.py` — injection at spawn, per provider

New helper `_skill_roots_for_project(project_ns)` — every configured path of
the active project (`lead_context._allowed_project_roots`) plus
`config.REPO_ROOT`, i.e. the exact same roots
`settings_window._new_role_skill_roots` scans, so the Skill Matrix and the
actual spawn-time injection always agree on what "exists".

Three call sites, one per provider branch already in `spawn()`:

- **claude** (teammate branch, where the role's `CLAUDE.md` appendix is
  assembled): appends `render_skill_appendix(base_role, roots,
  PROVIDER_REGISTRY["claude"].context_strategy)` to the same `_appendix`
  string that already carries project-memory/learned-notes/file-guards.
- **codex** / **gemini** branches: `ensure_agents_md()` gained a new `extra: str
  = ""` parameter (default empty — zero behavior change for every existing
  caller/test), appended after the base cheatsheet before writing. Both
  branches now compute `skill_policy.render_skill_appendix(base_role, roots,
  <provider>_spec.context_strategy)` and pass it as `extra`.

Both non-claude branches read `context_strategy` off `PROVIDER_REGISTRY`
(`provider_spec.py`) rather than hardcoding "codex/gemini get the
instruction-style block" — the decision of which bridge a provider gets is
now genuinely driven by its `ProviderSpec`, matching what phase 0 documented
but never wired up.

## Known gap (documented, not silently degraded)

`ensure_agents_md`'s `extra` content is **per-cwd, not per-role** — if two
DIFFERENT roles both map to a non-claude provider (`provider-roles.json`
override) and happen to share the same explicit `--cwd`, whichever spawns
last wins the AGENTS.md content, same as the base cheatsheet already behaves
today (one file per cwd, no per-role variant). This is the same tradeoff the
pre-existing "one manager + one marker" design already accepted for the base
file; it just now also applies to the skill appendix. In practice this rarely
bites since roles default to distinct cwds (frontend→web, backend→api, ...).
Noted directly in `ensure_agents_md`'s docstring rather than left implicit.

The other explicit gap is the one `render_skill_appendix` states in its own
disclaimer text for codex/gemini: a skill whose value comes from the Skill
tool's own execution (auto-trigger matching, bundled scripts the tool runs)
cannot be replicated by a plain markdown file pointer — only the
instructions/description text bridges over.
