# #100: MCP/plugin bridge for codex + agy panes

## Audit — what already existed before this task

| Piece | File | What it does |
|---|---|---|
| Claude MCP injection | `spawn_engine.py` claude branch | `browser_profile_mcp_config_path()` → `--mcp-config <path> --strict-mcp-config`. Role-aware allowlist (`pane_tools_policy.py`), per-pane browser-profile isolation for playwright/chrome-devtools (#39). |
| `pane-tools.json` | `pane_tools_policy.py` | Accepts `mcps`/`plugins` keys for **any** role name, including `codex`/`gemini` — but nothing ever read those keys for the codex/gemini spawn branches. Dead config for non-claude providers, exactly as issue #100 described. |
| `ProviderSpec.mcp_adapter_variant` | `provider_spec.py` (phase 0) | Field existed with values `claude="strict"`, `codex="none"`, `gemini="plugin_import"` — documented intent, zero call sites reading it (confirmed by grep before this task; same "planted but unwired" pattern phase 0 left for `context_strategy`, later wired by #103 phase 4). |
| codex/gemini spawn | `spawn_engine.py` codex/gemini branches | Built a minimal argv (binary + autonomy flags) with no MCP wiring at all. |

**Gap confirmed**: identical to the phase-4 skill-injection gap — a policy dimension (`mcps`) existed in config shape but had exactly one working call site (claude).

## Real-binary testing (2026-07-11, this machine)

Both `codex` (codex-cli 0.144.1) and `agy` (Antigravity CLI 1.1.1) are installed and on `PATH` — every claim below is empirical, not read from docs.

### codex: native per-session `-c` overrides — confirmed working

`codex mcp add` writes to the user's global `~/.codex/config.toml` — not
what we want (task spec explicitly says "ห้ามแตะ user global config"). But
the **top-level** `codex` binary (not just the `mcp` subcommand) accepts
`-c key=value` dotted overrides, documented as: *"Override a configuration
value that would otherwise be loaded from `~/.codex/config.toml`"*.

Verified live:

```
$ codex -c 'mcp_servers.echotest.command="node"' \
        -c 'mcp_servers.echotest.args=["-e","process.exit(0)"]' \
        -c 'mcp_servers.echotest.env={FOO="bar"}' \
        mcp list --json
```

→ `echotest` appears in the merged config (alongside the user's own
pre-existing `github` server), with `command`/`args`/`env` round-tripping
correctly, **including a Windows path with backslashes**
(`C:\Users\monch\test.exe` round-tripped exactly via `\\` TOML escaping).

Confirmed **zero mutation** of `~/.codex/config.toml`:
`grep -c echotest ~/.codex/config.toml` → `0` after the run. `-c` overrides
are genuinely request-scoped.

One caveat found and avoided: overriding a single nested key of an
**existing** global entry (`-c mcp_servers.github.enabled=false`) breaks —
codex errors `invalid transport in mcp_servers.github` because the
override table doesn't carry the entry's other required fields. Not a
problem for us: the bridge only ever **adds new, uniquely-named** servers
(same collision-avoidance the claude browser-MCP names already rely on),
never edits an existing user entry.

### agy: no MCP CLI flag (confirmed) — but plugin import genuinely bridges MCP servers, at global scope

`agy --help` has no `mcp`/`--mcp-config` surface at all — confirmed, matches
the issue. `agy plugin import claude` (bare keyword) found nothing on this
machine even though 8 real claude-plugins are installed
(`~/.claude/plugins/cache/...`) — none of them ship an `.mcp.json`, so
there was nothing to prove either way from real installed data alone.

Built a synthetic test fixture instead: a claude-style plugin dir with
`.claude-plugin/plugin.json` (name only) + a root-level `.mcp.json`
(`{"mcpServers": {"synthtest": {...}}}`, the real Claude Code plugin MCP
convention) + a `skills/` dir. Importing it **by explicit path**:

```
$ agy plugin import <path-to-plugin> --force
  [ok]    synthetic-mcp-test
          ✔ skills      : 1 processed
          - agents      : skipped (not found)
          - commands    : skipped (not found)
          ✔ mcpServers  : 1 processed
          - hooks       : skipped (not found)
Staged to C:\Users\monch\.gemini\config
```

`agy plugin list` afterward shows `"source": "claude-code"`,
`"components": ["skills", "mcpServers"]` — **the mechanism works**. The
staged file lands at
`~/.gemini/config/plugins/synthetic-mcp-test/mcp_config.json` with the
translated `mcpServers` entry. (Cleaned up with `agy plugin uninstall
synthetic-mcp-test` immediately after — this ran against the real dev
machine's live `agy` install, not a sandbox.)

**Why this isn't wired into spawn_engine.py despite working**: the staged
location is a **global, machine-wide** registry
(`~/.gemini/config/plugins/`), not scoped per-project, per-role, or even
per-cwd. Auto-importing a synthetic plugin on every agy spawn would:

1. Leak cockpit-controlled MCP servers (e.g. playwright/chrome-devtools)
   into the user's own non-cockpit `agy` sessions on the same machine —
   there is no session boundary the way codex's `-c` flags or claude's
   `--mcp-config` file pointer provide.
2. Collide across roles: two DIFFERENT roles both mapped to the `gemini`
   provider (`role-providers.json` override) would import into the SAME
   global plugin slot, last-spawn-wins — strictly worse than the already-
   documented one-`AGENTS.md`-per-cwd race (#103 phase 4's "Known gap"),
   since this isn't even per-cwd.
3. Persist across cockpit restarts and even across projects, needing its
   own uninstall/GC lifecycle management that doesn't exist yet.

Per the task's own instruction ("ถ้าไม่ได้ document gap ชัดๆ + graceful
no-op"), `mcp_bridge.py`'s gemini branch is a **documented no-op** — real
capability confirmed, deliberately not auto-driven until `agy` grows a
session-scoped MCP mechanism (a `--mcp-config`-style flag, or a per-cwd
`.agents/mcp_config.json`, mirroring what changelog 1.0.14 already hints
at with the migrated `config/mcp_config.json` path but which today is
still one file per machine, not per-session).

## What was built

### `mcp_bridge.py` (new)

Single adapter function `mcp_argv_for_provider(provider_name, base_role,
shard_idx, project_ns) -> list[str]`, dispatched on
`PROVIDER_REGISTRY[provider_name].mcp_adapter_variant`:

- `"strict"` (claude) — unchanged, byte-identical `--mcp-config <path>
  --strict-mcp-config` (or `[]`), still resolved from
  `shared_dev_tools.browser_profile_mcp_config_path`.
- `"session_override"` (codex, changed from `"none"`) — translates the
  SAME role's MCP server dict into `-c mcp_servers.<name>.<key>=<toml>`
  tokens (`command`/`args`/`env` only — codex has no `"type"` field, our
  shared JSON schema's `"type"` key is silently dropped).
- `"plugin_import"` (gemini) — documented no-op (see above).

Every variant resolves the SAME role→server dict via
`shared_dev_tools.browser_profile_mcp_config_path` — codex and claude
panes assigned to `qa`/`critic`/`designer` now get the identical,
browser-profile-isolated playwright/chrome-devtools set, just in each
provider's own wire format. A codex `qa` pane genuinely gains working
playwright/chrome-devtools MCP tools it never had before.

### `provider_spec.py`

- `codex_spec.mcp_adapter_variant`: `"none"` → `"session_override"`.
- Comments on both codex's and gemini's `mcp_adapter_variant` updated to
  cite the real 2026-07-11 test results above instead of the phase-0
  placeholder text.

### `spawn_engine.py`

Three call sites, one per provider branch (mirrors #103 phase 4's skill
injection pattern):

- claude branch: the old inline `browser_profile_mcp_config_path` +
  `--mcp-config`/`--strict-mcp-config` code replaced with a call to
  `mcp_bridge.mcp_argv_for_provider("claude", ...)` — same resulting argv.
- codex branch: `codex_argv.extend(mcp_argv_for_provider("codex", ...))`
  appended after the existing autonomy flags.
- gemini branch: same call for symmetry/documentation (always resolves to
  `[]` today per the gemini no-op above).
