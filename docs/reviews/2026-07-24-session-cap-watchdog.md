# Session-cap watchdog — design and implementation

Date: 2026-07-24
Audit input: `docs/qa-reports/2026-07-24-token-audit.md`, ranked reduction #1

## Why this exists

The audit replay found 6,462 requests above a 200k prompt ceiling across 84
sessions, with 828,764,173 excess prompt tokens in seven days. The repeated
cost came from carrying a long session into later requests, especially in
persistent Lead panes. The cockpit already reads the latest Claude
`last_usage.prompt` for each display-backed pane, so the watchdog consumes that
existing sample instead of adding another transcript scan or model request.

## Policy

- Default cap: 180,000 prompt tokens.
- Configuration precedence:
  1. `TAKKUB_SESSION_CAP_TOKENS`
  2. Qt setting `session_cap/prompt_tokens`
  3. built-in default `180000`
- Invalid and non-positive values are ignored rather than silently disabling
  the guard.
- A crossing is edge-triggered: the first sample at or above the cap warns;
  later samples above the cap do not warn again.
- A sample below the cap re-arms the watchdog. Transcript rollover (`/clear`),
  detach, and a newly attached session also reset the latch.

## Role-specific action

### Teammate panes

The cockpit emits a UI notice and queues a short advisory into the teammate.
Delivery polls for the pane's ready prompt and has no blind-timeout fallback,
so it never pastes while the current turn is running. The advisory tells the
teammate to finish and report its current task normally, then use `/compact`
before more work or allow the cockpit to reopen a fresh pane after
`takkub done`.

If the pane is replaced, exits, or its prompt drops below the cap before the
advisory is delivered, the queued delivery is cancelled.

### Lead pane

Lead receives a status-bar and system-tray notice offering `/compact` or a
clean session restart when safe. The watchdog never writes into Lead's PTY,
never submits `/compact`, and never closes/restarts Lead automatically. The
user retains the decision because Lead is the live command surface.

## Provider boundary and known gap (#103)

The meter and watchdog are gated by
`ProviderSpec.supports_token_meter` at `AgentPane.attach_session`. Claude is
currently the only provider with a compatible JSONL usage schema, so Codex,
Gemini, OpenCode, Kimi, Cursor, and Shell panes do not poll for usage and
cannot trigger this watchdog. This also prevents a non-Claude pane from
accidentally selecting an unrelated Claude JSONL under the same working
directory.

`HeadlessPane` records the provider capability on its shared model, but it does
not currently own a JSONL poller; therefore session-cap notifications are a
display-cockpit feature for now. Future providers/headless mode can opt in
only after they expose a normalized prompt-usage reader and set
`supports_token_meter=True`.

## Implementation map

- `session_cap.py`: threshold parsing and teammate advisory text.
- `agent_pane_model.py`: provider capability plus the edge-trigger/reset latch.
- `agent_pane.py`: consumes the existing usage sample and emits one cap signal.
- `spawn_engine.py` / `headless_pane.py`: propagate provider identity and
  enforce capability gating.
- `orchestrator.py`: role routing and ready-prompt-only teammate delivery.
- `main_window.py`: user-facing status/tray notice.
- `tests/test_session_cap_watchdog.py`: threshold, reset, provider, Lead, and
  no-mid-turn delivery coverage.
- `docs/architecture/depgraph.json`: regenerated by the repository hook for
  the new `agent_takkub.session_cap` module.

The implementation uses Qt timers, environment variables, QSettings, and
provider metadata already used by the application; it introduces no
platform-specific process or filesystem behavior.
