## A3 Root Cause Detail

The draft guard works for the normal live queue: typed bytes enter `_on_pane_input`, Lead state becomes `NONEMPTY`/`UNKNOWN_NONEMPTY`, and `_pump_lead_notify` refuses to write while `_lead_can_accept_injection()` is false.

The escape hatch is durable done-notice liveness:

1. A done notice is queued/spilled while Lead cannot accept injection.
2. `_reap_pending_done_notices()` sees the Lead alive but not flushable.
3. After `_DONE_NOTICE_STALE_S = 60.0`, it calls `_force_deliver_done_notices()`.
4. `_force_deliver_done_notices()` writes the payload and schedules verified Enter without checking `_lead_can_accept_injection()`.

This exactly matches the user report if the user draft remains unsubmitted long enough while a pane reports done. It is an intentional test-backed behavior, not just a missing bracketed-paste case.

## A7 Reuse Recommendation

Reuse A1's path convention and file writing, but do not mutate `_task_handoff_pointer` into a ledger API. Add a new `task_ledger.py` or similar with:

- `create_assignment(project, role, cwd, task, provider, effective_provider) -> ledger_entry`
- `mark_done(project, role, status, note_ref/evidence_ref)`
- append/update `index.md` or `ledger.jsonl` in `runtime/tasks/<project>/<date>/`

Then let `_task_handoff_pointer` become a consumer of the ledger file path for long/short tasks, while `done()` updates status before/around `_pane_state.pop()`.
