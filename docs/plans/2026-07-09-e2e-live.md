# Live E2E — Remote PWA HTTP API (2026-07-09)

Real HTTP requests against the running cockpit process (PID 28284), not unit tests.
Remote server: `http://127.0.0.1:9999/<secret_path>/...` (from `~/.takkub/remote.json`, `mode=control`, `password_hash` set).
Auth stack confirmed live: secret-path segment → bearer token → (for every route except `/api/verify-password`) `X-Session` from a verified password.

**Password caveat:** `remote.json` only stores a PBKDF2 hash — no plaintext available, and this is the user's real always-on cockpit (a real mobile client may be paired against it), so no brute-force attempts were made (would risk tripping `lockout_after_fails=5` and locking the real device out for up to 5 min backoff). One single wrong-password probe was sent (safe — 1 fail, threshold 5) to confirm the reject path, then stopped. Every route that requires an `X-Session` therefore stops at `403 password_required` and its business logic (payload shape, 400-on-bad-project, etc.) is **SKIPPED** below — documented per-item, not silently omitted.

## Conclusion on auth (item 5, expanded)
The "unauthenticated → rejected" claim is verified live, not just by unit test: every one of the 6 data/control routes tried returned either `404` (bad/missing bearer, bad secret path) or `403 password_required` (bearer ok, no session) — never a 200, never leaked data. This matches the documented design (`_reject()` gives a bare 404 that never distinguishes "wrong secret path" from "wrong token", and `_check_password_gate()` is the third factor gating every route but `verify-password`).
