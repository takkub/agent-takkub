# External agent patterns for Takkub Cockpit (Codex research)

**วันที่ค้น:** 2026-07-10  
**มุมวิจัย:** pattern จากระบบ multi-agent / coding-agent ที่โดดเด่นในปี 2026  
**ขอบเขต:** idea harvest เท่านั้น — **ยังไม่กรอง/จัดลำดับเพื่อตัดทิ้ง** เพราะจะมี critique round ต่อ  
**North-star ที่ใช้เทียบ:** _one command in → auto plan / divide / test / summarize_

## วิธีอ่านผล

- รายการ 12 ข้อด้านล่างเป็นการ **adapt แนวคิด** ให้เข้ากับ local-first, live-pane และ multi-provider ของ Takkub ไม่ใช่การ copy API/UI ของระบบต้นทาง
- `Effort`: S = ไม่เกินหนึ่งโมดูล/ผิว UI เล็ก, M = หลายโมดูล, L = เปลี่ยน orchestration หลัก, XL = ต้องมี runtime/persistence model ใหม่
- `Risk`: ความเสี่ยงเชิง product/architecture ไม่ใช่โอกาสเกิดบั๊กอย่างเดียว
- `Fit 1–5`: ความตรงกับ north-star; ไม่ใช่ priority score และ **ไม่ได้ใช้ตัดข้อใดออก**
- เลือก 5 ระบบหลักจาก “ความต่างของ pattern + มีเอกสารต้นทางตรวจสอบได้” ไม่ได้อ้างว่าเป็นอันดับ market share 1–5

## Baseline ของ Takkub ที่ไม่นับเป็น gap

Takkub มี Lead–Specialist, parallel/fleet, plan-first sharding, worktree isolation, QA-last, verify-fail feedback routing, task ledger, screenshot evidence scan, event log, role memory และ token **context-occupancy** meter แล้ว (`CLAUDE.md`, `README.md`, `src/agent_takkub/task_ledger.py`, `token_meter.py`, `role_memory.py`, `routing_planner.py`). ดังนั้นข้อเสนอข้างล่างจงใจขยับเลยจาก capability เหล่านี้ เช่น “replay จาก checkpoint” ไม่ใช่เสนอ task ledger ซ้ำ และ “structured memory” ไม่ใช่เสนอไฟล์ role-memory เพิ่มอีกไฟล์

ข้อจำกัดปัจจุบันที่มีผลต่อการ adapt:

- token meter อ่าน context occupancy ของ Claude session แต่ยังไม่ใช่ cross-provider session budget/cost controller (`token_meter.py:1-10`, `:194-240`)
- role memory เป็น Markdown ที่ agent append แบบ free-form; ยังไม่มี provenance/confidence/expiry/approval schema (`role_memory.py:115-137`)
- Claude panes ถูก spawn ด้วย `--dangerously-skip-permissions`; policy ของแต่ละ provider ยังไม่เป็น abstraction เดียวกัน (`spawn_engine.py:1487-1494`)
- task ledger และ `events.log` เป็นหลักฐานที่ดี แต่ยังไม่ใช่ deterministic execution checkpoint ที่ fork/replay เฉพาะขั้นได้ (`task_ledger.py`, `orchestrator.py`)
- มี stuck watchdog เชิง process/output แล้ว แต่ยังไม่วัด semantic non-convergence เช่น diff/test failure เดิมวนซ้ำ

## 5 ระบบหลักที่ดึง pattern

| ระบบ | Pattern ที่ยืนยันจากแหล่งต้นทาง | เหตุผลที่เด่นสำหรับ Takkub |
|---|---|---|
| **Claude Agent SDK / Anthropic coordination patterns** | generator–verifier พร้อม criteria/iteration cap/fallback; parallel evaluators + aggregation; orchestrator, agent team, message bus, shared state; subagent context/tool isolation | ใกล้ Lead–Specialist ที่สุด และให้กรอบยกระดับ quality/termination โดยไม่ต้องเปลี่ยน Takkub เป็น framework ใหม่ ([coordination patterns](https://claude.com/blog/multi-agent-coordination-patterns), [workflow patterns](https://claude.com/blog/common-workflow-patterns-for-ai-agents-and-when-to-use-them), [SDK subagents](https://code.claude.com/docs/en/agent-sdk/subagents)) |
| **LangGraph** | checkpoint ทุก graph step, durable resume, pending writes, human interrupt, replay/fork และแยก thread state ออกจาก cross-thread Store | เป็น reference ที่ชัดที่สุดสำหรับ replay/debug และ memory lifecycle ([persistence](https://docs.langchain.com/oss/python/langgraph/persistence), [time travel](https://docs.langchain.com/oss/python/langgraph/use-time-travel), [interrupts](https://docs.langchain.com/oss/python/langgraph/interrupts)) |
| **OpenHands SDK** | append-only event log + auto-persist conversation, fork, auditable condensation event, risk-scored confirmation policies, persisted stuck-detection state | เหมาะกับ local-first และให้ pattern ที่แยก audit history, working context, security policy ออกจากกัน ([persistence](https://docs.openhands.dev/sdk/guides/convo-persistence), [fork](https://docs.openhands.dev/sdk/guides/convo-fork), [condenser](https://docs.openhands.dev/sdk/arch/condenser), [security](https://docs.openhands.dev/sdk/guides/security)) |
| **Cline** | task-level token/cost/time + Git checkpoints/history, per-subagent budgets, multi-model routing, auto-compact ที่ย้อนกลับได้, progressive-loaded skills, OpenTelemetry usage events | ให้ product pattern ที่ผู้ใช้ “เห็นและคุม” ต้นทุน/context ได้ ไม่ใช่ observability หลังบ้านอย่างเดียว ([tasks](https://docs.cline.bot/core-workflows/task-management), [subagents](https://docs.cline.bot/features/subagents), [model orchestration](https://docs.cline.bot/cli/samples/model-orchestration), [skills](https://docs.cline.bot/customization/skills), [OTel events](https://docs.cline.bot/enterprise-solutions/monitoring/opentelemetry-events)) |
| **Devin / Devin Review** | diff-aware test plan + recorded evidence, codebase-aware PR review, confidence/severity findings, auto-review on new commits, per-session ACU hard caps, MCP marketplace auditability | เด่นด้าน “agent ทำเสร็จแล้วพิสูจน์ให้คน review ได้เร็ว” ซึ่งต่อยอด QA/evidence ของ Takkub โดยตรง ([testing flow](https://docs.devin.ai/work-with-devin/computer-use), [Devin Review](https://docs.devin.ai/work-with-devin/devin-review), [2026 release notes](https://docs.devin.ai/release-notes/2026)) |

ระบบที่ตรวจประกอบแต่ไม่ได้เป็น 5 แกนหลัก: **CrewAI** (Crews vs deterministic Flows, guardrails/memory/observability), **AutoGen** (multi-agent conversation + configurable human-in-the-loop) และ **Cursor Background Agents** (isolated async agents + environment snapshots). Pattern ส่วนใหญ่ซ้ำกับของที่ Takkub มีแล้วหรือถูกระบบหลักข้างบนอธิบายละเอียดกว่า จึงใช้เป็น cross-check ไม่ได้แปลว่าระบบเหล่านี้ด้อยกว่า ([CrewAI](https://docs.crewai.com/), [AutoGen](https://autogenhub.github.io/autogen/docs/Use-Cases/agent_chat/), [Cursor background agents](https://docs.cursor.com/background-agent)).

## ความสัมพันธ์กับ north-star

ไอเดียทั้ง 12 ข้อประกอบกันเป็นชั้น ไม่ใช่ feature list ที่ต้องสร้างพร้อมกัน:

```text
one command
  → budget/risk-aware route (1, 9)
  → durable plan + resume (6)
  → agents coordinate on typed evidence (10)
  → context/memory survive handoffs (7, 8)
  → diff-derived tests + bounded verification (2, 3, 4)
  → semantic termination (11)
  → evidence-linked summary
  → replay/fork when the orchestration itself fails (5)

role/skill marketplace (12) expands what each node can do without hardcoding it.
```

จุดสำคัญคือ north-star ไม่ควรวัดแค่ “auto” แต่ต้องวัดว่า run นั้น **หยุดถูกเวลา, พิสูจน์ completeness ได้, resume ได้, และอธิบายค่าใช้จ่าย/ความเสี่ยงได้** ด้วย

## คำถามสำหรับ critique round ถัดไป (ยังไม่ตัดสินแทน)

1. ใน 12 ข้อ อะไรควรเป็น product surface ที่ user เห็น และอะไรควรเป็น engine invariant ที่ไม่มี toggle?
2. Execution checkpoint (ข้อ 5–6) ควรเริ่มจาก orchestration events เท่านั้น หรือรวม provider transcript/tool calls ตั้งแต่แรก?
3. Budget ของ CLI subscription ควรนิยามเป็น token-equivalent, wall-clock, rate-limit pressure หรือหน่วยผสม เพื่อไม่แสดงค่าเงินปลอม?
4. Quality contract/judge panel ควร trigger จาก risk tier แบบใด เพื่อไม่ทำให้ quick fix แพงกว่างาน?
5. Risk-adaptive policy จะบังคับ parity ขั้นต่ำอะไรได้จริงเมื่อ Claude/Codex/Gemini เปิด hook/permission surface ไม่เท่ากัน?
6. Memory และ blackboard ควรใช้ store เดียวกันแต่ต่าง namespace หรือแยก trust boundary คนละระบบ?
